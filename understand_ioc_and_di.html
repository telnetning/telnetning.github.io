理解 IOC 与 DI

在 Spring 框架中，有两个核心概念贯穿始终，即控制反转（IOC）和面向切面编程（AOP）。这里主要是我对 IOC 以及其主要实现方式依赖注入（DI）的理解。

  • DI 的实现

普通方式与简单工厂模式

对于面向对象编程，尤其是 Java 这种纯面向对象的编程语言，一般功能的实现都是由多个类共同完成。在一个类需要使用另外一个类，很常见的，我们需要在该类中引用另外一个类。比如：

我们在类 A 中需要使用到类 B，我们一般会这么写：

public class A {
    public void importantMethod() {
        B b = new B...  //get an instances of B
        b.usefulMethod();
    }
}

即，要使用 B，我们需要在类 A 中引用类 B。若 B 是一个具体类，我们可以直接 new 一个 B 。但是若 B 是一个接口呢？我们可能就需要考虑 B 的各种具体实现。

结果就是：A 的可重用性大大降低。

为此，我们考虑使用工厂模式来生产 B 的实例。为此，我们定义一个砍树的任务。主要的类是 Person 和 Axe。最普通的方式如下：

public class Person {
    public void main(String[] args) {
        Axe axe = new Axe();
        axe.chop();
    }
}
public class Axe {
    public void chop() {
        System.out.println("Chop tree with axe");
    }
}

这里使用了两个类，主角是 Person 类，Person 类调用 Axe 类来进行砍树这一任务。

这种在 Person 中直接 new 一个 Axe 的模式很不好。为什么呢？考虑实际情况，Axe 可能是一个抽象的概念，实际上，我们手上拿的可能不是 Axe， 而是一把 StoneAxe 以及一把 SteelAxe。那么我们怎么选择使用哪一把 Axe 呢？

容易想到的就是在 Person 的 main 方法中，要用哪个 Axe，就 new 哪个 Axe。

结局就是我们上面说到的，A 的可重用性大大降低了。因为 A 和 Axe 的具体类耦合在一起了，而其具体类 StoneAxe 和 SteelAxe 的创建实现方式可能差异很大。

Java 提倡的一个准则就是面向接口编程，作为 Person 类其实不应该关注 Axe 类的具体实现。Person 类只需要干两件事情：

  • 说明我需要哪种 Axe
  • 调用 chop 方法

而所有的具体实现，包括如何创建一个 Axe，都不应该是 Person 类应该关心的。Person 类关心的，只是 Axe 的接口。

所以，好的方式是，定义一个 Axe 接口，它负责创建具体 Axe 以及定义 Axe 的各种操作。这里，接口有两种定义方式，一种是抽象类，一种是接口。这句话很拗口，实际上，前一个接口是一个概念，指的是面向接口编程这种概念，即各个组件之间合作和沟通的桥梁。后一个接口（即 Java 中的 interface）是和抽象类（abstract class）相对的实现方式。至于究竟是选择抽象类（abstract class）还是接口（interface），这又是另外一个话题了。这里采用抽象类作为父类。

public abstract class Axe {
    abstract void chop();
}

然后定义其子类：

public class StoneAxe extends Axe {
    public void chop() {
        System.out.println("Use stoneAxe to chop tree.");
    }
}
public class SteelAxe extends Axe {
    public void chop() {
        System.out.println("Use steelAxe to chop tree.");
    }
}

在 Person 类中指明需要的 Axe ，然后调用 chop 方法，而生产具体的 Axe 这一过程，就可以交个另外一个单独的类来完成：

public class AxeFactory {
    public static Axe getAxe(String axeType) throws Exception {
        if(axeType.equals("stoneAxe")) {
            return new StoneAxe();
        }   else if(axeType.equals("steelAxe")) {
            return new SteelAxe();
        } else {
            throw new Exception();
        }
    }
}

Person 中只需要说明所需要的 Axe 的种类，然后调用 chop 即可。

    public class Person {
        public static void main(String[] args) throws Exception {
            Axe axe = AxeFactory.getAxe("stoneAxe");
            axe.chop();
            Axe axe = AxeFactory.getAxe("steelAxe");
            axe.chop();
        }
    }

这个时候，Person 类就与 Axe 的具体实现无关了，即解耦了。Person 类不用管如何创建一个 Axe 类，每当需要一个 Axe 的时候，Person 类就向 AxeFactory 请求一个。这就是简单工厂模式，AxeFactory 中的 getAxe 方法就是一个静态工厂方法。

IOC 与 DI

说了这么久的简单工厂模式，这和 IOC 以及 DI 有什么关系呢？

上面的例子中，无论是普通方式还是基于简单工厂模式，都是在解决一个类（Person）对另一个类（Axe）的依赖问题。

简单工厂模式的优势在于，它保证了 Person 类只与 Axe 接口耦合，而不和 Axe 的具体的实现类耦合。这是符合面向接口编程原则的。Person 类同时引入了另外一个问题，就是 Person 类与 AxeFactory 类的耦合。

这种一定程度上的耦合其实无可避免。简单工厂模式的作用也是其优势就是在于，将 Person 类与 Axe 具体类的耦合从 Person 自身转移到了 AxeFactory 的静态工厂方法中去了。

试想，如果在一个大型项目中，当类逐渐增多，类间的关系越发复杂，那么，我们是不是需要创建更多的工厂方法？工厂方法中内容（if-else）也将变得越来越繁琐。

所以，我们考虑将这种耦合关系（Person 与 Axe 的具体实现类）再次转移，转移到哪儿？转移到框架中，转移到配置文件中。

于是，Spring 的 IOC 就横空出世了。

Person 类依旧不需要自己 new 一个 Axe，也没有了工厂方法，一切都交给了 Spring。Spring 负责创建对象，并且把对象注入到 Person 类中。

这个时候，我们写 Person 的代码，完全不需要考虑 Axe 它从哪里来，它要怎么来，只管用就行。

所以，Person 的代码会是像这样的：

public class Person {
    private Axe axe;

    public void setAxe(Axe axe) {  //use setter to injection
        this.axe = axe;
    }

    public void useAxe() {
        axe.chop();
    }
}

尽管提供了setAxe 方法，但是这个方法并不是我们自己的代码调用的。实际上，我们要完成人砍树这一动作，现在只需要调用 Person.useAxe() 即可。代码如下：

public class Test {
    public static void main(String[] args) throws Exception {
        ApplicationContext ctx = new
                ClassPathXmlApplicationContext("beans.xml");  //This sentence will init all singleton bean

        Person p = ctx.getBean("person", Person.class);

        p.useAxe();
    }
}

我们得到一个 Person 对象之后，直接调用了 useAxe 方法。而 useAxe 中的 axe 从哪里来，是哪个 axe，都不需要我们在代码中写明。

这就是 Spring 中的 IOC。

IOC，Inversion of Control，翻译为控制反转。什么意思？之前是 Person 对象主动去要一个 Axe 实例（不管是普通方式还是简单工厂模式）。而在 Spring 中，Person 对象中的 Axe 实例并不是主动获取的，而是被动接受的。该 Axe 实例实际上是由框架在合适的时候注入到 Person 中去的。所谓的反转，指的就是，对象（这里是 Person 对象）获取它的协作对象（这里是 Axe 对象）的方式反转了，由主动索取变成被动接受了。

主动变成被动，是 IOC 的主要内容。

Spring 中的协作对象（Axe）是注入到对象（Person）中去的。也就是对象（Person）依赖（Dependency）框架（Spring）来注入(Injection)协作对象（Axe）。

所以，Spring 中的这种 IOC 方式，也被称作依赖注入（Dependency Injection，即 DI）。

所以，DI 是 IOC 的一种实现方式，IOC 和 DI 并不是同一个概念，虽然在 Spring 中常常混用。

总结一下，Spring 利用框架，将对象与其依赖对象的耦合关系转移到了配置文件中。在运行过程中，Spring 根据配置文件，在恰当的时候，向对象中注入其协作对象。这就是 Spring 中的 IOC/DI 的核心功能。

DI 的实现

下面，我们将 Spring 中的 IOC 以及 DI，统称为 DI。

在 Spring 中，最主要的配置依赖方式是依据配置文件。上面例子中的配置文件如下：

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd">

    <bean id="person" class="Person">
        <property name="axe" ref="stoneAxe"></property>
    </bean>

    <bean id="stoneAxe" class="StoneAxe" />
</beans>

Spring 是一个大容器，其中的对象都称之为 bean。上面 xml 文件中配置了两个 bean，分别为 person 以及 stoneAxe。在 person 的配置中，利用 property 配置了其依赖的协作类 stoneAxe。

当然，除了 xml 文件之外，Spring 也提供了 基于注解的配置方式，但是如今 xml 配置依旧是主流。

DI 的实现方式有一下三种：

  • 基于 setter 方法注入（设值注入）
  • 基于构造注入（构造注入）
  • 基于接口注入

在 Spring 中采用了 setter 方法注入以及基于构造的注入。

上面的 xml 配置就是以 setter 方法注入的。

    <bean id="person" class="Person">
        <property name="axe" ref="stoneAxe"></property>
    </bean>

    <bean id="stoneAxe" class="StoneAxe" />

基于 setter 注入，其命名原因在于，框架给 Person 对象注入，需要 Person 对象提供 setter 方法。

    public void setAxe(Axe axe) {  //use setter to injection
        this.axe = axe;
    }

这样，Spring 在实例化一个 Person 实例之后，会立即调用该实例的 setter 方法，向该 Person 实例中注入一个 Axe 实例。

基于构造器注入，xml 文件中使用 constructor-arg 指定依赖类。

    <bean id="person" class="Person">
        <constructor-arg ref="stoneAxe"/>
    </bean>

    <bean id="stoneAxe" class="StoneAxe" />

指定了 StongAxe 为构造器的参数。而 Person 同样地，需要提供一个构造器交给框架调用：

public Person(Axe axe) {
    this.axe = axe;
}

Spring 默认会调用类的无参数构造器，但是每一个 constructor-arg 就表示一个带参数构造器及其参数。Spring 调用带参数的构造器，即可初始化注入协作实例。

反射和 DI

DI 实现了在运行过程中，产生实例并注入。这正是利用了 Java 的反射机制。

首先考虑框架如何生成 Person 实例呢？它只需要读取配置文件，获取到 bean 的名称和对应的类（class 指定），然后，获取该类的 Class 对象，再调用 newInstance 即可：

String idRef = ....; //从 xml 文件中读取，bean 的 id 指定
String classRef = ....; //从 xml 文件中读取，bean 的 class 指定
Class clazz = Class.forName(classRef);
Object obj = clazz.newInstance();

//存储到 Spring 容器中
container.put(idRef, obj);

现在生成了 person 对象，那么如何完成基于 setter 注入呢？利用反射获取到 Person 实例的 setter 方法，然后调用即可。

String nameStr = ....; //property 中的 name 属性，
String refStr = ....; //property 中的 class 属性
String setterName = "set" + nameStr.substring(0,1).toUpperCase() + nameStr.subString(1);  //根据 name 得到 setter 方法名
Object paramBean = contain.get(refStr);  //从容器中 get 一个协作类实例

setter.invoke(paramBean);  //调用 setter 方法注入

同样地，考虑构造器注入，和基于 setter 的注入其实原理一致，只不过一个调用的是 setter 方法，一个调用的是构造方法。

其它

以上说的是Spring 中的 DI 的核心概念和原理，Spring 的 DI 为了方便用户使用，实际上还围绕这些基本概念和原理对 bean 的操作提供了许多功能。包括：

  • 提供多种创建 bean 的方式（静态工厂、实例工厂等）
  • 给 bean 提供了多种作用域（scope）（singleton、property，request，response，global）
  • bean 的语法简化配置（基于XML Schema、SpEL）
  • bean 的逻辑复杂配置（嵌套 bean，bean 的继承等）
  • bean 注入及销毁前后的行为定义(init-method、destroy-method)
  • 基于依赖类型不同的 bean 的定义方式(注入集合等)
  • ......
