Linux内核课堂笔记（6）

李龙海老师的 linux 内核课程的第六课的课堂笔记。主要内容：运行时堆。

今天讲程序运行时内存分布的最后一部分：堆内存。我们应该还记得，堆内存是从地址低端向高端增长。先说一下堆内存的基本内容：

  • 在需要时主动申请
  • 堆内存在程序主动放弃之前一直保持有效
  • 占据了进程地址空间的绝大部分，对于大的缓冲区，不建议使用全局变量，更不建议使用局部变量，最好使用堆内存
  • 由 c 运行时环境（C Runtime Environment）负责管理，管理方法为 批发－>零售

批发和零售是什么意思？因为堆内存管理是由运行时环境管理的。程序向操作系统申请一段内存空间，这个时候，首先是由运行时环境向操作系统申请至少一般一次至少 4K（也就是一页）的内存。然后根据程序的需要，把申请到的内存分配一些给程序。没有分配完的空间留给程序未来申请。

我们看一下用于零售堆内存的C标准库函数：

  • malloc(size)
  • calloc(n, size)
  • realloc(ptr, size)
  • free(addr)

尤其需要关注的是 free 函数，一旦在申请了内存之后，新手都比较容易忘记在使用完成后使用 free 函数进行内存释放。

另外 linux 中提供了用于批发堆内存的系统调用。

  • brk(addr) //将堆尾指针 brk 调整到指定位置
  • sbrk(incr) //将堆尾指针 brk 增加或减少指定数值
  • mmap

这里提到了一个概念，叫做系统调用。系统调用和一般函数的区别在于，一般函数都是运行在用户态，而在系统调用中，函数会从用户态转到内核态。

在从用户态转到内核态的过程中，耗时相对很长，这就是为什么堆内存的管理是批发－>零售策略。批发的次数越少，那么使用系统调用的次数就更少，从而节省时间资源。

那么堆内存的管理方法是怎么样的呢？挨着分配么？怎么记录哪些被分配了哪些没有被分配呢？

然后我们关注的是 free 函数带来的问题，也就是堆的紧缩问题。起始就是内存分配的逆运算，在两个表上进行计算，倒也不算麻烦。经过两个 free，我们的两张表会回到初始状态。

除了双表法外，还有其它方法用于堆内存分配的管理。比如空闲链表法，就是把未使用的内存块使用链表串联起来，这样分配内存和 free 内存就是对链表的删除和插入操作。此外，还有位图法、对象池法以及伙伴关系算法等。

我们说过，堆内存是使用内存过程中最容易出现问题的地方，下面是使用 malloc 以及 free 的一些注意事项，每一个程序员都应该牢记在心的：

  • 刚刚分配的动态内存的初始值是不确定的。
  • 不能对同一指针(地址) 连续两次进行 free 操作。
  • 不能对指向静态内存区(全局变量)或栈内存区(局部变量)的指针应用 free (但可以对空指针 NULL 应用 free)。
  • 对一个指针应用 free 之后，它的值不会改变，但它指向了一个无效的内存区，这时称该指针为“悬空指针”。
  • 如果没有及时释放某块动态内存，并且将指向它的指针指向了别处，就会造成“内存泄漏”。
  • 执行 malloc 和 free 函数有一定的代价，所以对于较小的变量不应该放在动态内存之中，并且尽量避免频繁地分配和释放动态内存。

上面的注意事项不能理解地记住，后果就是下面的：

  • 内存分配未成功，却使用了它。
  • 内存分配虽然成功，但是尚未初始化就引用它。(误认为初始值为 0 )
  • 内存分配成功并且已经初始化，但操作越过了内存的边界。
  • 忘记了释放内存，造成内存泄露。
  • 释放了内存却继续使用它。

这些都是十分常见的错误。一个程序员，必须对其所使用的内存了如指掌。使用内存前，问问自己，这块内存存在么，这块内存有意义么，程序写完前，问问自己，有没有哪些内存没有释放？

我们再来说说悬空指针，什么叫悬空指针，就是不为 NULL 且没有指向有效地址的指针。解引用悬空指针，会产生不可预知的后果。
看个例子：

void somefunction(){
    int *p;
    ......
    *p = 7;
    ......
}

这里指针 p 在未初始化就赋值。结果就是p是一个悬空指针，它并不指向一个有效的地址。正确的做法是，在声明指针 p 的时候，对 p 进行初始化：

int *p = NULL;

另外就是在 free 一个指针之后，继续访问该指针。

int main()
{
    int *p = NULL;
    p = (int*)malloc(sizeof(int));
    *p = 5;
    free(p);
    ......
    *p = 7;
    ......
}

对 free(p) 之后，其申请的内存已经被释放，这个时候再来使用其指向的内存，后果不可预计。

另外，内存泄漏也是十分重要的命题。几乎是程序员最爱犯的毛病之一。有的时候，程序员尽力避免，却依然会犯这个错误。不信看看下面这段代码：

void MyFunction(int nSize)
{
    char *p = new char[nSize];
    if(!SomeFunc()){
        printf("Error");
        return;
    }
    
    ...//using the string pointed by p
    delete p;
}

这个程序员还特意在最后用了 delete 来进行内存的释放。但是呢，由于程序的逻辑关系，在 if 中，代码是有可能退出的，也就是第 6 行的 return 语句。一旦 return 了，那么 p 指向的内存呢？并没有进行释放！内存泄漏！

所以说，为什么有些编程规范中，只允许程序员在函数的最后一句 return。也就是不允许程序有其它出口，只能有一个入口一个出口。否则中途退出的话，程序员一不注意，就内存泄漏了。

另外一个容易产生内存泄漏的情况我们在前面也说到过，就是团队协作的时候，或者你去使用第三方代码（库）的时候。看下面：

char *TranToEng(const char *inputStr)
{
    //将中文翻译成英文
    char *outputStr = (char *)malloc(...);
    ......
    return outputStr;
}

int main()
{
    char *chineseStr＝"欢迎光临";
    char *englishStr = TranToEng("欢迎光临")；
    printf("%s", englishStr);
}

这段代码的意思是，调用了一个第三方的函数，用来将一段中文翻译成英文，函数返回的结果是翻译后的英文。问题在哪里呢？由于只有第三方的函数能知道翻译之后的字符串的长度，所以英文字符的空间是由第三方函数来申请的。然后该函数将字符串返回给主函数。主函数使用了这个字符串。这个时候，编写主函数的人，很难想到需要去释放这个字符串的内存空间，因为这段内存并不是他申请的。

这种情况下调用第三方函数的程序员，应该变得足够敏感，意识到这块内存的释放是自己的责任。主动执行 free 函数，还有就是查看第三方的文档，一般第三方会提供自己的 free 函数来供使用者调用。

内存泄漏实在是太容易犯的一个错误了。如何避免内存泄漏？

其一就是运行检测法。

  • 定义自己的 malloc 和 free 函数，或者对 new 和 delete 进行重载，在运行时跟踪内存的分配和释放情况。
  • 利用专用的检测工具，如 BoundChecker，Purify 和P reformance Monitor，我一般用的是 valgrind，在这里也推荐一下。

其二就是使用一些程序设计技术。

  • 智能指针技术。主要思想就是，当栈回收一个指针的时候，先回收该指针指向的内存。
  • 为 C++ 增加垃圾回收机制

我们还有一点东西要说的，就是关于 malloc 的代价问题。前面说过 malloc 的代价相对较高，一般小的缓冲区就不需要用堆了。那么 malloc 的代价在哪里？

我们先说说中断和异常的区别。中断会等待当前指令执行完，而异常会切换到新的指令，新的指令执行完后重新执行当前命令。

当我们使用 malloc 的时候，如果上次批发申请的内存已经不够了，那么我们需要调用系统调用，这是一个付出代价的地方。

另外一方面，当使用系统调用批发申请一块内存的时候，系统只是口头承诺给了一块内存，实际上这个时候还没有给内存。什么时候真正给内存呢？访问内存的时候。怎么实现的？就是使用异常。当指令去访问一块内存，我们以为自己已经有了这块内存，实际上还没有，这个时候就会导致异常，异常结果就是真正地分配内存，然后重新执行访问内存的指令。这个过程中又有一次系统调用。

最后，我们看一下前面说过的 task_struct，也就是那个描述进程的结构体。我们关注结构体中对内存分布的描述：

  • task_struct 用于描述一个进程， task_struct 中有一个字段：struct mm_struct *mm
  • mm_struct 用于描述一个进程虚拟地址空间的布局。 mm_struct 中有一个字段 mmap，指向了一个链表：struct vm_area_struct *mmap
  • mmap 链表中的一个节点 vm_area_struct 记录了实际分配的一个内存区域

task_struct 的内容大致如下:

struct task_struct {
/* these are hardcoded - don't touch */
  volatile long        state;          /* -1 unrunnable, 0 runnable, >0 stopped */
  long                 counter;
  long                 priority; 
  ... 
/* filesystem information */  struct fs_struct     *fs;
/* open file information */  struct files_struct  *files;
/* memory management info */
  struct mm_struct     *mm;
  ... 
}

结构体中有一个 mm 成员，指向 mm_struct 结构体，mm 的意思是Memory manage。那我们再看看 mm_struct 结构体：

struct mm_struct {
    struct vm_area_struct   *mmap;
    struct rb_root          mm_rb;
    struct vm_area_struct   *mmap_cache;
    ...
    atomic_t                mm_users;
    ...
    unsigned long           start_code;  //代码区起始位置
    unsigned long           end_code;    //代码区结束位置
    unsigned long           start_data;  //数据区起始位置
    unsigned long           end_data;    //数据区结束位置
    unsigned long           start_brk;  //堆区起始地址
    unsigned long           brk;        //当前堆指针
    unsigned long           start_stack;//栈区起始位置
    unsigned long           arg_start;  //参数区起始位置
    unsigned long           arg_end;    //参数区结束位置
    unsigned long           env_start; //环境变量起始位置
    unsigned long           env_end    //环境变量结束位置
}

上面注释过的字段，我们可以看出内存分布。至于 vm_area_struct 结构体，我们这里先不讲，以后再说。

注意一下这里的地址，都是用 unsigned long 类型表示的，内核中都是这么干的。
