linux内核课堂笔记（7）

李龙海老师的 linux 内核课程的第七课（并不是实际的第七课，中间的有大概五次课是关于 linux 的内存管理，一来这部分比较晦涩而且复杂，二来我也还没有从中提取到一条好的脉络来整体理解记忆，所以之后再总结，噢，但愿我的拖延症不再作祟！）的课堂笔记。这节课笔记的主要内容：进程和线程，linux 中的线程，进程状态，fork 函数。

进程和线程是 CS 专业的人听到的最高频词汇之一。我们这里就来说说进程和线程。

首先我们不说 linux，说说一般的进程的概念。当代码还只是硬盘上的文件的时候，代码只是代码，但是一旦代码呗加载到内存中运行，代码就不只是代码了，由代码会产生一系列的东西，这就是进程。可以外行地讲，进程就是跑起来的代码。也就是，进程是代码的实例。显然地，我们可以让一份代码产生多个实例，只要我们再次运行一个这个代码就可以了。所以进程和代码还有一个多对一的关系。

然而，进程不只是运动着的代码，为了让操作系统能管理进程，进程为了能和其它进程通信，进程为了能够有地方放东西，等等，围绕着进程的，就是一整套资源，这些资源在 linux 系统中，有 PID，task_struct，独立的进程地址空间，信号函数对应表，打开的文件描述符等等。所以在回答面试小哥进程的概念的时候，我们常常会回答：进程是资源分配的实体。

那么再说到线程，这里我们做一个比喻。某司有一个项目要做，嗯，这就是我们的静止的程序啦，它现在就是一个老板桌子上的一个项目计划书。老板说，ok，让我们干起来吧。于是，老板制定了一个项目负责人，然后，该负责人拉起了一帮人，然后给员工分配了工位，还给每个员工配备了交流使用的水果机，把大家拉进了一个 XX 讨论组……好吧，有些牵强，但是这些都是其实都是为了和进程真正拥有的各种资源进行类比。现在从老板角度看来，负责人管理着的是，就是公司的资源，老板并不在意内部操作。那么，这个时候，项目组，就是一个进程。

然后我们扯回项目内部，项目可能需要多个项目成员做同样的工作，也可能需要多个项目成员拥有不同的技能，做不同的工作，然后成员间进行衔接交流。这是两种比较常见的方式。这里，每一个人基本都是并行工作的，每一个人都是一个独立的执行线路。是的，这就是我们所的线程。所以，线程是代码运行的最小调度。各个线程也是可能干着同样的事情，也可能干着不同的事情然后需要进行线程间的通信。

线程在进程内部，如同个人在项目组内部。项目组的资源对于个人是部分共享的，线程对进程的资源也是部分共享的。也就是，各个线程有自己的非进程内共享的资源。

上面这个又臭又长的比喻，在 linux 下可能更加显得有些不合适了。为什么？因为，linux 中没有真的线程！什么意思？linux 中的线程是用进程模拟的。程序员以为自己在用线程，实际上用的是进程。linux 就这样 fool 了程序员。在看 linux 是怎么 fool 程序员之前，我们先来看看，linux 的进程是怎么一回事。
linux 下，我们说的任务就是进程。一个任务就是一个进程。我们来看看进程占有的资源。

进程拥有 PID，就是唯一标志，也就是进程的 ID 号。每一个进程都有一个进程 ID，我们常常会用 kill num 来杀死一个进程，这里的 num 就是进程号。

然后，重要的就是进程描述符，也就是 task_struct。这是一个结构体，它记录了进程的所有信息，所以它很大。另外，我们也称其为进程控制块（PCB）。

PID 信息毫无疑问也是在 task_struct 中，也就是 pid 字段。另外我们知道 linux 中所有进程组成一个进程树，每一个进程都有父进程（除了 init ），getpid 以及 getppid 分别可以用来获取自己的 pid 以及父进程的 pid。

其实，在前面我们已经提到过这个 task_struct 结构体了，它大概长这个样子：

struct task_struct{
    volatile long state;
    void *stack;
    atomic_t usage;
    unsigned int flag
    ……
}

其实看起来和一般的结构体没有什么不同。因为它就只是结构体啊。但是，它里面的每一个字段的意义都比较重要，整个进程的所有信息都在这个结构体了。然后，操作系统把所有进程的 task_struct 串联起来了，嗯，这个就是我们前面提到过的内容了。总而言之，这个结构体，描述了进程，并且控制着进程。

接下来比较重要的就是独立的进程地址空间了。我们已经知道进程有 4G 的独立地址空间。其中有各种各样的东西，包括堆啊栈啊只读区域啊这些。这些也是进程拥有的资源。

通过以上这些，我们知道，创建进程，其实就是建立一个 task_struct，然后创建独立的进程地址空间（其实也就是我们还没有提到过的一套独立的页表）。

我们来整体看一下进程和线程的关系的一张图：

这里的 PCB 就是进程控制块，也就是 taski_struct 结构体。TCB 就是线程控制块。我们前面说同一进程内的各个线程共享进程的很多资源，比如打开一个文件的句柄，再比如，堆。但是线程为了自己能够好好地跑程序，相互之间不乱套，它们也有自己的私有资源，比如栈。假设你这儿线程跑着一个函数，函数的局部变量都在栈中，你肯定不希望别的线程来访问你的栈甚至改变它。

还有一个不容易弄清楚的概念，就是 TLS，意思是 Thread Local Storage，翻译成中文就是线程局部存储。为啥要有线程局部存储？现在是这样的，在进程中声明一个全局变量，这个时候进程中的各个线程都可以访问这个变量，这便于各个线程之间进行数据交换，但是也给线程的同步带来了麻烦。如果反之，在线程的函数中定义一个变脸，这个变量又只能在该函数内部访问。那么如果我们想要一个在且仅在线程内部能够全局共享的变量呢？

如上图，我们想要的就是线程1中的变量i，这个变量只在该线程内可见，但是在该线程内的各个函数都可以访问这个变量。这个时候，我们需要的就是线程局部存储技术。具体的线程存储技术在各个操作系统中是不一样的，所以具体实现这里也不多讲。

那么我们接下来讲讲 linux 是怎么用进程来模拟线程的，linux 中的线程叫做轻量级进程。

其一，模拟线程首先要解决的就是地址空间共享的问题。也就是各个线程之间都可以共同访问的内存空间。在linux的内存管理中我们学过，进程的地址空间就是它指向的一套独立的页表，我们要两个进程之间共享内存地址空间，说需要做的就是，让它们指向同一套页表。

对于一个进程，它有自己的一套页表，也就是自己的内存地址空间。当它调用创建线程的时候，它创建了一个进程（就是一个 task_struct），这就是进程 2，然后呢，进程 2 的页表直接指向了进程 1 的页表。这个时候，进程 1 和进程 2 就拥有了同样的内存地址空间。也就解决了共享资源的问题。

其二，线程需要有自己独立的栈区，这个更简单。在前面我们讲过，栈区其实主要就是由 EBP 和 ESP 决定，我们需要做的，就是每创建一个进程时候，更改其 ESP 和 EBP 的值就行了。

其三，当我们杀死进程的时候，进程中的所有线程都应该被杀死。这个稍微思考一下，还是上面的图，我们把所有的这个进程 1 通过调用 ThreadCreate 弄出来的进程给一个记录不就行了么，当杀死进程 1 的时候，我们把所有的相关进程就杀掉。嗯，是的，linux 就是这么干的。还是在 task_struct 中，由这样两行：

gid_t gid;
gid_t tgid

这里不仅有 gid，还有 tgid，tgid 什么东西，thread group id，线程组 id，上面这些进程属于同一线程组，拥有相同的 tgid，所以当杀死进程 1 的时候，也就可以通过 tgid 找到其它的轻量级进程，杀掉。

通过以上这些黑魔法，linux 就把进程当作线程来使了，所有 linux 中的线程叫做轻量级进程。linux 就是这样 fool 程序猿的。

接下来要说的是 linux 中的进程地状态，linux 中的进程状态为下面五种：

  • TASK_RUNNING
  • TASK_INTERRUPTIBLE
  • TASK_UNINTERRUPTIBLE
  • TASK_STOPPED/TASK_TRACED
  • TASK_ZOMBILE

我们看一张图，来看看进程的整个生命周期：

图中说明了进程在各个状态之间的转换关系，基本完全符合人类的认知，运行，睡眠，等待信号，暂停，死亡等，所以这里不详细展开。但是还是有几个问题要说到：

一个是 TASK_INTERRUPTIBLE 和 TASK_UNINTERRUPTIBLE 的区别，竞争进入睡眠本来是等待资源唤醒的。但是唤醒进程的除了是进程预期的资源之外，还有可能是信号。这两者的区别是前者会对信号进行响应，而后者不会。想一想，为什么我们用 CRTL＋C 有时候能杀死一个进程，有时候却不能呢？

说到信号，我们最开始也提到过，进程内部的资源还包括一个叫做信号处理函数表的东西。我们知道系统的中断有中断向量表，这个信号处理函数表和中断向量表的功能相似。它记录了信号，以及信号对应的函数地址。收到信号，然后进入对应的函数。

还有一个比较重要的就是，TASK_ZOMBIE，僵尸态，也是“将死态”，这个说来话长，我们搬个板凳坐下慢慢说。
首先我们来说说父进程和子进程。当我们写程序的时候，我们都会有一个 main 函数，它长这个样子：

int main(int argc, char **argv)

这次我们关注下 main 函数的返回值，也就是这个 int，它返回给谁了？理论上，它返回给了调用它的人，main 函数是干活用的，干完活了，活干成什么样了，总得汇报一下吧，就是通过这个 int 返回值汇报的。那么它的调用者是谁，父进程！所以，main函数的返回值返回给了它的父进程，那么问题来了，当一个进程结束的时候，它的资源被释放了，它都存在了，它如何给父进程返回值？

这里 linux 又使用了取巧的做法。当进程结束的时候，也就是你的 main 函数 return 了之后，开始回收资源了。但是该进程的资源都被回收，除了 task_struct 这个结构体。也就是子进程结束时 task_struct 并没有被回收，子函数的返回值也放在了 task_struct 中。那么子进程结束后，父进程通过子进程的 task_struct，就能拿到子进程的返回值。然后父进程回收该 task_struct 结构体，这下才彻底回收了子进程。

所以这里你可以看到，在子进程结束和子进程被彻底回收之前，存在一个中间状态，这个状态就是要死还没死的状态，形象地被称为僵尸态。

我们可以再多关注一下。父进程是如何回收子进程的 task_struct 的呢？如果写过 fork 的同学，应该会用过 waitpid 这个函数，在 fork 后面，我们都会有一个对应的 waitpid:

pid = fork()
//in parent process
waitpid(pid,……)

这里的 waitpid 就是用来干这个事情的，它读取子进程的 task_struct，得到其返回值，然后回收子进程的 task_struct。而且我们可以思考一下这个 waitpid 的工作方式……

另外，如果我们不调用 waitpid 呢？很多人都会有这个毛病，如果在写一个服务程序，结果可能就是不断产生僵尸进程，最后，终究有一天，系统会挂掉。

还有一个值得提的，如果父进程在 waitpid 之前就跪了，那么子进程怎么办？子进程就永远没有人回收了么？事实是，如果发生这种情况，那么子进程会被托管给 init 进程，也就是 linux 进程的老祖宗。这个时候，这个进程，就是我们说的“孤儿进程”。而 init 进程会定期处理这些孤儿进程。

最后我们说说 fork，这个 linux 中几乎最有存在感的函数。最开始学 fork 的人，都会迷惑它的用法，但是相信大家都已经很熟悉这个函数了。

fork 的迷惑之处在于，fork 调用一次，却返回了两次。为什么返回两次呢？因为从 fork 返回的前一刻开始，已经多了一个进程，代码已经多了一个运动着的分身。原来的进程和新的进程几乎一模一样（因为 fork 的操作只是创建了一个 task_struct，一套与原进程相同的页表），不同的是它们的进程 id（PID）。所以 fork 函数返回的就是 pid 相关的。在父进程中，它返回了子进程的 pid，在子进程中，它返回了 0。这样后面就能知道哪个是父进程哪个是子进程了。

这里回顾一下，子进程为什么几乎和父进程一模一样，而不做更多的贴合自己的初始化工作。这是因为，我们提到过的写时拷贝。

我们这里来看一个题目，特别经典的：

int main()
{
    int i;
    for(i=0;i<3;i++){
        fork();
        printf("-");
    }
    return 0;
}

首先我们知道最后肯定是有 8 个进程了，肯定有人用树状图已经画出来了，有 14 条枝，也就是会有 14 次的 printf，那么应该打印出 14 个“_”。

但是实际运行一下，你就会发现，它居然打印了 24 个“-”。为什么！？

实际上，这里跟缓冲区有关，当 printf 打印的时候，如果没有刷新缓冲，那么 printf 的东西先在缓存中，并没有实际打印到 stdout 中。重点是，我们的 fork 函数！fork 的子进程继承了父进程的几乎所有东西，也包括缓冲区，所以在子进程中，尽管它还没有 printf，它已经缓存了父进程 printf 的东西，所以父进程的一次 printf，实际在子进程中会多存在一份。这里，最后的 8 个进程，它们缓冲中，都会有 3个“-”，当程序结束的时候，它们就被全部刷新到 stdout，也就是屏幕上了。
但是我们如果将 printf 那一句改成：

printf("-\n")

那么就能在 printf 的时候直接打印到 stdout 中，结果就是 14 了。
