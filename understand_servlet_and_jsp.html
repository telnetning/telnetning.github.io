理解 servlet 与 jsp

Web 的本质是 C/S 架构，client 端，一般就是浏览器，负责向服务器端发送请求，并且渲染显示服务器端返回的 html 文件。而服务器端的职责就是接受客户端的请求，返回内容给客户端。返回的内容一般就是 html 文件。

但是服务器端仅仅返回静态的 html 文件是不够的，为了实现完善的业务，动态 html 生成几乎是必须的，比如动态显示当前商品的数量。这就是静态网站和静态网站的区别。像我的博客这种，就是一个静态博客。博客内容就是许多的 html 文件组成。像评论这种需要动态生成的，其实是用的第三方工具，并不是博客本身的。

动态博客为了动态的生成 html 文件，最常用的就是使用脚本去和数据库交互，然后生成一个 html 页面。在 LAMP 技术栈中，用 apache 作为 web 服务器，用 mysql 作为数据库，用 php 作为脚本来生成 html 文件。apache 接受用户的请求，将请求以及环境变量转发给 php 脚本，php 据此处理请求，必要的话操作数据库，然后将生成的页面返回给 apache，apache 再将页面返回给客户端。apache 和 php 之间的接口就是我们常说的 CGI 接口，php 脚本在这里就被称作 CGI 脚本。

什么是 servlet

上面说的是 php web 开发的原理。本来 CGI 脚本可以使用几乎任意语言来实现。但是 CGI 本身存在着很多缺陷，java web 并没有采用 CGI 来构建 web 应用，而是使用了 servlet 的方式。servlet 本身是 java 中的一个类，类中有如下一些方法：

  • init(ServletConfig)
  • getServletConfig()
  • getServletInfo()
  • service(servletRequest, servletResponse)
  • destroy()

servlet 就是一个类，web 容器可以调用这个类给客户端提供响应。init 方法就是给web容器调用的。web 容器调用 init 之后即初始化了 servlet 类，并且该类将一直提供服务，直到 web 容器destroy 这个类。servlet 工作流程为：

  1. 创建实例
  2. 调用 init 初始化
  3. 使用 service 响应客户请求
  4. 调用 destroy 回收资源，销毁实例

需要注意的是，一个 servlet 被创建后，将一直存在容器中，给客户端提供服务，直至容器决定销毁实例。这和原始 CGI 是不一样的，原始 CGI 被人诟病的原因之一就是它采用 「fork and exec」 的方式，每一次响应请求都需要创建实例然后销毁，效率很低下。所以后来才有了 FastCGI 之类的改良设计。servlet 和 FastCGI 一样，都是实例会多次提供服务。

不管是什么设计，最基本的流程都是 web 服务器将 request 转发给脚本，然后获取脚本返回的 resopnse。同时，威尔伯服务器需要向脚本提供一些环境变量，在 CGI 设计中，web 服务器是通过进程的环境变量或者管道传送环境变量。在 servlet 设计中，servlet 的 getServletConfig 也可以完成这些工作。servlet 中住主要的响应函数是 service ，web 容器通过调用它来向客户端提供服务。

servlet 实例

根据 servlet 的工作原理，我们可以自己写一个 servlet 来试试。安装好 tomcat 之后，在 tomcat 的 webapps 目录下创建一个 helloworld 的目录，作为应用名。然后创建 WEB-INF 目录，再在此目录下创建 src、lib、classes 目录，分别放置源文件、库文件以及编译之后的 class 文件。

如下编写 servlet：

import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.*;
import java.io.PrintStream;
import javax.servlet.annotation.WebServlet;

@WebServlet(name="firstServlet", urlPatterns={"/firstServlet"})
public class FirstServlet extends HttpServlet
{
    public void service(HttpServletRequest request, 
            HttpServletResponse response)
        throws ServletException, java.io.IOException
        {
            PrintStream out = new PrintStream(response.getOutputStream()); 
            out.println("<h1>This is the first Servlet.</h1>");
        }
}

这里我们写了一个 servlet，它继承了 HttpServlet 类。而 HttpServlet 通过 extends GenericServlet 类从而 继承了 Servlet 类，因为 GenericServlet 类 implements 了 Servlet 类。

在第8行我们使用 annotation 的方式来申明这个 servlet 的监听地址，另外一种方式是在 tomcat 的配置文件中配置。这里使用 annotation 配置这个 servlet 监听 /firstServlet 这个 url。然后我们在类中重写了 service 方法，即使用该方法来提供服务。然后我们手动编译该类。

javac FirstServlet.java -classpath ../../../../lib/servlet-api.jar

其中的 classpath 选项是指定 tomcat 中的 servlet-api.jar 包。否则编译时候是找不到 javax.servelt.* 这些包的。编译成功之后把生成的 FirstServlet.class 文件放到 helloworld/WEB_INF/classes 中去。然后打开 tomcat 就可以看到结果了。

结果就是我们在 FirstServlet 中打印的结果。这样一套完整的 http 请求过程就完成了。同时，该类也可以连接数据库，从而动态生成 html 页面。另外，该类中的 service 也可以替换成 doGet 方法，因为 HttpService 也提供了 doGet 以及 doPost 方法。而 service 默认会采用 doGet 或者 doPost 方法来响应用户的对应请求。

servlet 的局限以及 jsp 的产生

这样手工写 servlet 固然可以，但是明显存在很多局限。

  • 每次修改一个 servlet 都必须手动重新生成 class 文件，重启 tomcat 来保证 重新加载该 servlet。
  • servlet 中需要手工 print 大量的 html 元素。业务逻辑和表现层没有分开。

这就像 php 本身也可以 echo 出一个网页来，但是为了开发的方便以及前后端的写作，最后还是利用在 html 文件中写 php 的方式。servlet 的本质就是在 java 中写 html，因此，为了使得 html 和java 尽可能分离，jsp 出现了。jsp 的本质是，在 html 中写 java。当然你也可以直接写静态的 html，这比在 servlet 中 print 出来一个 html 页面要简单得多。网页中的大多数元素是不变的 html 元素。所以 jsp 中在 html 中写 java 的方式，比在 java 中 print 大量 html 的方式要简单得多，并且，更有利于前端人员对 html 页面的开发。

一个简单的 jsp 页面如下：

<%@ page contentType="text/html; charset=utf8" language="java" errorPage="404.jsp" %>

<html>
<head>
    <title>欢迎</title>
</head>
<body>
    Hello, Jsp.    
    <% 
    out.println("Hello, world!");
    %>
</body>
</html>

就像一个 html 文件一样，然后我们就可以直接访问该网页。而且，我们可以在其中插入 java 代码，就像我们用 php 写网页时候在 html 文件中插入 php 代码一样。

从该 .java 文件中我们大概可以看出来，该类是通过 _jspService 提供服务的，那么为啥不是 service 方法呢。我们看看 HttpJspBase 的源码。其中有这么一段：

public final void service(HttpServletRequest request, HttpServletResponse response) 
    throws ServletException, IOException 
{
    _jspService(request, response);
}

…………

public abstract void _jspService(HttpServletRequest request, 
        HttpServletResponse response) 
throws ServletException, IOException;

原因就是 HttpJspBase 中重写了 service 方法，该 service 方法只是调用了 _jspService 方法。所以只需要实现 _jspService 方法，然后该方法即可提供服务。jsp 文件被转成 java 文件，实际上就是自动实现了该方法。所以我们其实可以完全绕开 jsp 文件，然后自己写一个类，实现该方法，也可以提供服务，但这样也就失去了 jsp 的意义了。这里只是想说明 jsp 的工作原理。

从 jsp 到 servlet 的过程并不需要程序员的参与。程序员只需要写 jsp ，然后在浏览器中访问就行了。其他都由 web 服务器在背后搞定。

如图显示了 jsp 的生命周期。

  • tomcat 将其翻译成 java 文件
  • java 编译器将生成的 java 文件编译成 servlet 类型的 class 文件
  • 运行环境运行该 class 文件

jsp 文件就是为了简化 web 页面的编写而产生的。

另外一个需要注意的是，jsp 在什么时候被 tomcat 翻译呢？对于程序员来说，修改了 jsp ，并不需要做什么重启之类的，直接访问，网页就会发生变化。为什么呢？

如下：

  • 在第一次访问 jsp 的时候，该 jsp 被翻译
  • 在 jsp 发生改变的时候，jsp 会被重新翻译

而这些是自动的，不需要程序员干涉。但是需要注意，因为这种特性，在第一次访问一个 jsp 文件或者在修改某 jsp 之后访问时，访问速度会比较慢，因为此时该 jsp 需要被重新翻译成 serlvet。

jsp 的使用

理解了 jsp ，再学习 jsp 的使用会相对明白很多。jsp 中需要理解的几类东西主要是：

  • 编译指令
  • 动作指令
  • 内置对象

编译指令顾名思义，就是在 jsp 编译成 servlet 过程中起作用的指令。分别是：page、include、taglib。page 可以用来设置当前页面的一些指令，include 可以静态包含其它文件。taglib 的重要性在于，进一步简化 jsp 的写法。跟很多框架使用的标签库其实是一类东西，主要是用来简化前端人员写后端代码的过程。

动作指令指的是运行时的动作指令。比如实现页面间的跳转（forward），页面的内容包含（include）等。其中有一个比较特殊的是 useBean，使得可以在 jsp 中实例化一个对象。其实对于一个后端开发者来说，可以直接在 jsp 中写 java 就行了，发明 useBean 这样的东西，应该还是为了前端人员的考虑吧。

比较重要的就是 jsp 中的内置对象了。因为 jsp 最终就是一个 servlet，那么它应该和 servlet 一样，能够对一些环境变量和 session 之类的东西进行操作。需要理解的是，这些东西本身就是 servlet 中有的，所谓 jsp 内置这些对象，就是规定了一种写法，你在 jsp 中写一个 application，那么它就会被翻译成 servlet 中的 application 对象。这些对象一共有 9 个，分别为：

  • application
  • config
  • exception
  • out
  • page
  • pageContext
  • request
  • resopnse
  • session

如果注意的话，你就会发现，它们都是上面 jsp 翻译成的 servlet 中的 _jspService 方法的变量或者参数。但是可能你没找到 exception，实际上也是可以有的，就是当页面被设置成 errorPage 的时候。所以，jsp 所谓的内置对象，实际上就是 _jspService 方法中的对象。其中 request 和 response 分别代表客户端的请求以及对客户端的回复。out 可以代替 response 输出字符流。page 代表页面本身。exception 在页面为 ErrorPage 的时候起作用，可用来打印错误信息。config 可以用来获取一些配置信息。而 application、page、request、session 则是代表了不同的变量共享范围。

你可以在 jsp 中使用它们，如果能够理解它们都是 _jspService 中的变量，由 _jspService 的话，那么可能会更好地使用它们。
