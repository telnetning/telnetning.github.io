Linux内核课堂笔记（3）

李龙海老师的 linux 内核课程的第三课的听课笔记。涉及内容：linux 下 C 程序的静态布局与动态布局、volatile 等关键字作用。

关键字以及变量生存期

程序中的变量生存期：我们把程序运行时一个变量占有内存空间的时间段称为该变量的生存期。分为：静态、自动以及动态三种。

  • 静态生存期：全局变量都具有静态生存期，它们的内存空间从程序开始执行时就进行分配，直到程序结束时候才被收回。
  • 自动生存期：局部变量和函数形参一般都具有自动生存期，它们的内存空间在程序执行到定义它们的复合语句（包括函数体）时才分配，当定义它们的复合语句执行结束时内存被收回。
  • 动态生存期：具有动态生存期的变量的生存时间是由程序员自由控制的，其内存空间用 new 操作符分配，用 delete 回收。

当定义局部变量时，可以加上 auto、static、register 来指出它们的生存周期。

register 关键字的作用：使用寄存器存储。

下面是关 volatile 关键字的内容。

有这样一段程序：

int flag;
void onInterupt()
{
    flag = 0;
}

void thread1()
{
    flag = 1;
    while(flag){
        //do something 
    }
}

这样一段程序，意图比较明显，就是在 thread1 这个线程运行的时候，别的线程可以通过调用 onInterupt 函数来使线程终止循环。

但是这段程序是有问题的。原因在于 thread1 这个线程中，while 循环每次都会读取 flag 的值，但是在 while 循环内部并未对 flag 变量有任何操作，这个时候编译器就会对这段代码进行优化，优化结果可能会是，将 flag 放到寄存器中，while 循环后面读取 flag 的时候直接从寄存器中读取，而 onInterupt 改变的 flag 的值是内存中的 flag，这样就会导致 while 不会停止，无法产生中断的效果。

这样的多线程问题比较容易产生这样的问题，这个时候，volatile 就能派上用场了。volatile 变量是告诉编译器这个变量可能会被意想不到地改变，这样编译器就不会去假设这个值，优化器在使用这个变量时候都会小心地重新读取，而不是使用保存在寄存器中的备份。

下面是另外一个关于 volatile 使用场景的例子：

假设有一硬件设备，硬件工程师给了某一地址，这里我们不讨论物理地址和虚拟地址，假设转换后的虚拟地址为 0x40000000，然后有这样一段程序：

char *p = (char *)0x40000000;
*p = 5;
x = *p;

这样一段代码有问题么，咋一看起来不会有问题，其实当然是有的。gcc 会对其做优化，结合第二行和第三行，gcc 可能直接就让 x=5，这样看起来也没什么问题？真没问题么？
考虑其使用场景：

在这样的使用场景中，寄存器中的值是可能会被外设和 cpu 改变的。外设和 cpu 之间可能 恰恰就是使用寄存器中的值进行通信。所以上面代码的第二行和第三行的意思是给地址赋值，然后读取可能被改变的地址。比如 cpu 运行第二行后，把这个地址的值设置为 5，然后读取地址，看看外设有没有改变这个值。

这儿，这个地址的值也是易变的，在其前面放上 volatile 关键字，就可以很好解决这个问题。

还有另外一个 volatile 的使用场景：在 C 代码中嵌入汇编代码。

__asm__(".align 2\n\t"
        "movl %eax., $ebx\n\t" 
        ......
        "subl %ecx, $ebx")

汇编中的代码大都是对寄存器的直接操作，这样的代码编译器也会对其做处理，一个明显的例子就是寄存器可能对你要求使用的 A 寄存器改成使用 B 寄存器，这本身不会有问题，但是一旦有别的线程在使用 B 寄存器，这个时候就会冲突导致寄存器的状态不可定了。为了告诉编译器不要做类似于寄存器替换这样的处理，我们可以加上 volatile 关键字，像这样：

__asm__ __volatile__(......);

另外一种可能的解决方法就是，使用某个寄存器前，将其值存储起来，使用完该寄存器后，把该寄存器的值重新 push 回去。

同时，gcc 发明的一套复杂的汇编语言可能有可以解决这个问题。

程序的静态布局与动态布局

考虑这样的两段程序：

int a;
int b=5;
int c;
int d=6;
int e;
int main()...
int a;
int c;
int b=5;
int d=6;
int e[100];
int main()...

问 a,b,c 等全局变量的地址在运行期间是否会改变？----不会。

问程序 1（前面的）中的 a,b,c,d,e 等变量在内存中是否为连续存放的？----a,c,e 是连续存放的，b,d 是连续存放的，但是他它们之间不是连续存放的。

问两个程序哪个生成的可执行文件更大？------实际上是一样大的。

为什么呢？

b,d 初始化过，它们的初始值需要记录，a,c,e 等未初始化的值，则只记录了它们一共需要的内存大小。

还是不太懂，接着往下看。

int main()
{
    char *str = "Hello, World!";
    str[0] = 'h';
    printf("%s\n", str);
    return 0;
}

这段代码的运行结果是什么呢？

实际上，会出现错误。为什么？这里 str 在程序运行时处于常量区，理论上不应该被修改。

但是我们如果这么写：

char str[] = "Hello, World!";

则不会有任何问题，为啥？这个时候 str 是一个局部数组，运行时被放在栈中。

同理，下面这段代码：

char *str1 = "Hello World";
int gval = 10;
int main(){
    char *str2 = "Hello World";
    printf("%u, %u \n", str1, str2);
    return 0;
}

运行结果又是什么呢？----结果是两个字符串的地址是一样的！

两个字符串都是放在常量区，并且值相同，所以编译器直接让它们指向同一个地方了。

C 程序的静态布局指的是 C 程序编译之后生成的可执行文件。C 程序的动态布局指的是 C 程序运行时候的内存动态布局。

C 程序的可执行文件遵循 ELF 文件格式规范的。ELF 的文件中分为许多 section，不同的 section 被装载到运行时虚拟地址空间不同的区域（VMA），不同的 VMA 具有不同的属性，主要包括三个方面：是否可读，是否可写，是否可执行。

静态布局主要包含以下的几个 section：

  • .text：C程序生成的汇编指令
  • .init：主程序执行前执行的代码
  • .finit：主程序结束之后执行的程序
  • .data：已初始化的全局变量
  • .bss：未初始化的全局变量
  • .rodata：只读数据区

分析下面的程序中的变量都在哪个 section:

int g1 = 84;
int g2;
const int g3 = 100;
char *str = "Hello!";
void f1(int i){ printf("%d\n", i); }
int main()
{
    static int var1 = 85;
    static int var2;
    int a = 1;
    int b;
    char *p = (char *)malloc(1000);
    f(a+b);
    return 0;
}

g1 在 .data 区，g2 在 .bss 区，g3 在 .rodata 区，str 在 .rodata 区，var1 在 .data 区，var2 在 .bss 区，%d这个字符串也是在 .rodata 区的。至于 i,a,b 以及 p 这几个，得在运行时内存布局说。

这里来看看装载器装载对 elf 和 VMA 做的映射：

运行时内存大致分为以下几类：

  • 保留区（Reserve）,操作系统预留放置一些信息。
  • 代码区（CODE VMA)，由 ELF 的 .text 指令 section 映射过来。
  • 只读 VMA，由 ELF 的 .rodata 节映射过来，放置的是一些只读的数据。
  • 数据区（DATA VMA），数据区，由 ELF 的 .bss 以及 .data 映射过来，放置的是一些变量数据。
  • 堆区（HEAP VMA），手动分配的内存都是从堆里分配。
  • 栈区（STACK VMA），局部变量等由栈区来管理。
  • 堆区栈区之间的内存空间，堆区往上增长，栈区往下增长，一旦相遇，说明内存用完了，其中还存放着程序需要的动态链接库。
  • 栈区与内核空间（0xc0000000以上）之间的预留区，放的是程序运行的环境变量，main 函数的参数之类的。
  • 可以注意一下写在图最右边的各个区的权限情况。

然后，我们来看一下一段代码：

void getmemory(char *p)
{
    p = (char *)malloc(100);
    strcpy(p, "Hello, world");
}

int main()
{
    char *str = NULL;
    getmemory(str);
}

这段代码有问题么，明显是有问题的。这里的 str 是实参，而 p 是形参，形参和实参的地址不是一样的，对形参进行地址分配，实际上对实参并没有起到作用。

那么要怎么办呢？

首先我们知道，C 中的所有参数都是传值引用。当我们传递给函数一个指针的时候，实际上是函数拷贝了一份指针，使得形参指针指向和实参指针相同的地方，用数组做参数就是这么干的。

也就是形参只能改变实参指向的内存的值，但是并没有办法改变实参指向的内存地址。

在子函数中修改主函数中的指针的指向是永远没有用的。

所以上面这段代码的问题就明白了，str 这个实参指向 NULL，在子函数中，给形参分配地址，也就是更改了指向，实际上对实参不会有任何作用。

解决方法呢？把申请的地址 return 回来是可以考虑的一种方式。还有一种方式就是传递给子函数一个指向指针的指针。

假设申明为 char **str，那么 str 以及形参 p 的值都是 *str，也就是另外一个指针，依旧的，形参指向 *str 这个地址也是不能改变的，但是形参可以改变 *str 的值，也就是 *str 的指向，这个时候，也就等于改变了实参指向的指针指向的内存地址。

C 语言中的参数既可以向里传值，也可以用来向外面返回值，这就是 C 语言的入参和出参的含义。

另外，C 语言中的内存管理是比较重要的，多人协作的时候有可能会出现一个人申请的内存需要由另外一个人去释放这种情况。这个时候读懂别人的 API 是比较重要的。
