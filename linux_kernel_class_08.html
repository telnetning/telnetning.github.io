上次讲到了 fork 函数，父进程将几乎的资源都复制给了子进程一份。这里的资源几乎包含了你能想象到的和遇到的一切，包括寄存器的值。但是唯独 EAX 寄存器的值没有直接 copy 给子进程。为什么呢？

对于一个函数来讲，我们在前面讲过，它的返回值是存储在 EAX 寄存器中返回的。fork 函数同样不例外，但是 fork 函数在返回之后的代码执行时就有了两份拷贝，即，fork 函数会有两次返回，分别返回到父进程和子进程中。而根据返回值的不同，我们就可以判定哪一个是父进程，哪一个是子进程。所以返回给父进程和子进程的 EAX 寄存器的值是不同的。

下面我们再来看一个 fork 函数的应用例子：

int main(){
int fd1 = open(“data_file1”, …);
int fd2 = open(“data_file2”, …);
pid_t pid;
if((pid = fork()) ==0){ 
close(fd2);
read(fd1, buffer, len);
… // 处理文件1的数据     
}
else {
close(fd1);
read(fd2, buffer, len);
… // 处理文件2的数据
}    
}

这里比较有意思的地方是第 7 行的 close(fd2) 以及第 12 行的 close(fd1)，不理解 fork 容易感到迷惑：怎么把两个文件描述符都关闭了呢？

实际上是，两个 close 分别在子进程和父进程中关闭的。看看代码想要干的事情吧，就是用两个进程分别处理两个文件的内容。在第 2、3 行中，程序打开了两个文件，拥有了两个文件描述符，当执行 fork 的时候，两个文件描述符都被继承给了子进程。而我们想做的只是子进程处理 file1，父进程处理 file2。所以在子进程中我们其实已经不需要 file2 的文件描述符了，同样的，在父进程中，也不再需要 file1 的文件描述符。这就是会有两个 close 的原因。

我们可以看看在 windows 中是怎么样使用多进程的。windows 中不同于 linux 的做法，windows 的做法可能更符合人类的理解。当需要另外一个进程做一件事情的时候，那么主进程直接就再启动一个进程去执行程序。

windows 中的这个函数叫做 createProcess。

int main()
……
createProcess(“A.exe”);
createProcess(“B.exe”);
……
//然后主程序接着做自己的事情了
}

然而 windows 中，一般都是用线程来做多任务。

linux 中有点类似与 windows 的 createProcess 的函数是 exec。exec 函数用于创建一个新的进程，并且新进程以另外一个可执行程序为执行脚本。但是，一旦执行了 exec，原进程就“不存在”了，原进程被 exec 占领了，exec 占用了原进程的绝大部分资源，进程空间装入了新的程序。

exec 函数簇有多个函数，比如 execl，它的定义大致如下：

#include <unistd.h>
int execl(const char *path, const char *arg, ……)

path 和 arg 分别为可执行程序的路径以及参数 1，它可以接受多个参数。

exec 函数簇的执行效果通俗来理解就是，从一个可执行程序切换到了另外一个可执行程序。这个时候，我们就想了，如果我们想要保留原进程，也就是想要保留父进程怎么办？

结合上面的 fork，我们可以使用这样的方法，用 fork 造出一个子进程来，然后在子进程中执行 exec 簇函数，这样就能保证父进程依旧存在了。

就像是这样：

#include <unistd.h>
int main(int argc, char *argv){
if(fork()==0) 
　　if(execl("/bin/echo", "echo", "executed by execl",NULL)<0)
　　perror("Err on execl");
　　}
 // 父进程做其他事情
 return 0;
}

在 unix 时代，fork 和 execl 的这种配合使用是很常见的事情，但是在那个时代这样使用的效率很低。

原因是这样的，在那个古老的年代，是没有写时拷贝技术的！这意味着，fork 之后的子程序需要将父进程的所有资源一次性都拷贝到自己的进程地址空间。而在 fork 和 execl 配合使用的时候，这些拷贝实际上做了无用功。好不容易拷贝完了，马上就执行 execl，载入新的可执行程序，进程空间的资源直接就被覆盖了。

于是unix 时代，为了解决这个问题，于是 vfork 函数就应运而生了。很明显地，它和 fork 的不同之处就是它不复制父进程的资源，而是共享父进程的资源，直到碰到了 exec 函数，子进程才会从父进程复制部分资源。

另外，vfork 还有一个特性就是，vfork 之后，子进程一定比父进程先执行。在子进程 exec 或者 exit 之后，才会执行父进程。

#include <stdio.h>

int main(){
int num = 1;
int child;

if(!(child=vfork())){
    printf("child begin sleep.n");
    sleep(2);
    printf("child sleep over.n");
    printf("child_pid:%dn", getpid());
    printf("child_num:%dn", numexit(0);
} else {
    printf("parent_pid:%dn", getpid());
    printf("parent_num:%dn", num);
    }
}

执行结果如下：

child begin sleep.
child sleep over.
child_pid:10333
child_num:1
parent_pid:10332
parent_num:2

从执行结果我们可以看出，尽管子进程睡眠了，但是子进程中的代码依旧在父进程之前执行。另外，子进程中改变了 num 的值，对父进程也起作用了。这就是 vfork 和 fork 函数不一样的地方。

后来，到了 linux 时代，出现了写时拷贝技术，vfork 就用的很少了，只在极少数场合使用。

对于第一点性质，即子进程总是在父进程之前运行，并且直到子进程 exec 或者 exit 了，父进程才会执行，也就是父进程被阻塞了。这一性质，可以作为一个 shell 程序的核心。

shell 是这样的程序，它接受用户的输入，检查输入的合法性，然后用 vfork 执行输入。这个时候，shell 程序本身就阻塞了，直到用户输入的命令执行完退出。

而对于第二个特性，也就是子进程和父进程完全共享内存空间，常常会让程序员遇到奇怪的 bug 而不知所以。

由于子进程和父进程完全共享内存空间，包括栈空间。一个比较常见的错误是，我们在 vfork 之后，在子进程中调用 return。

return 和 exit 的效果是不一样的，return 会导致弹栈，回到上级函数执行。子进程在 return 的过程中也会调用 exit，这个时候父进程得到子进程退出的消息，也开始从 vfork 返回。但是这个时候，栈已经被子进程弹了，也就是栈已经被子进程弄乱了，不再是父进程认识的那个样子。父进程这个时候再往哪儿返回，就不知道了。

这个问题具体也可以参见 陈皓的博客 (http://coolshell.cn/articles/12103.html)。

最后我们再来说一说 file 结构体，我们第一个例子中使用过文件描述符，但是文件描述符究竟是什么呢？文件怎么就成了一个整数了呢？

当我们使用 fopen 打开一个文件的时候，返回值是 FILE * 类型，也就是一个指向 File 类型的指针，而 File 本身是一个结构体，c 使用 File 结构体来描述文件。

它的大致定义如下：

typedef struct  
{  
short level ;  
short token ;  
short bsize ;  
char fd ;  
unsigned flags ;  
unsigned char hold ;  
unsigned char *buffer ;  
unsigned char * curp ;  
unsigned istemp;   
}FILE ;

在结构体中存在着一个 char fd 的字段。这里的 fd 就是文件描述符，每当我们创建一个结构体的时候，也就创建了一个文件描述符，然后这个文件描述符被返回给了用户。这样做的好处显而易见，用户通过文件描述符来操纵文件，更加简单，同时也一定程度上保证了对文件操作的安全性。

可以注意到的是，这里的 fd 的类型为 char，所以 fd 的最大值不过是 255，也就是说系统同时能够打开的文件描述符也只有 255 个。另外，0、1、2 的文件描述符分别为标准输入、标准输出以及标准错误流。
