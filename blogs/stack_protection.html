栈保护机制

某神问了我一个问题，于是我趁机学习了一下栈保护的知识。
事情起源于我的 Linux 课堂笔记的第二课，其中有这样一段代码：

int add(int x, int y){
    return x+y;
}
typedef int (*FP)(int,int);
int main(){
    unsigned char buff[1024];
    unsigned char *ps = (unsigned char*)add, *pd=buff;
    void *pp = buff;
    while(1){
        *pd = *ps;
        if(*ps == 0xc3) break;
        ps++;pd++;
    }
    FP fp = (FP)pp;
    printf("3+5=%d\n", fp(3,5));
    return 0;
}

某神问我这段代码有没有跑过，他跑的时候有问题。于是我也跑了下，段错误。

使用 valgrind 检测下：

==29276== Process terminating with default action of signal 11 (SIGSEGV)
==29276==  Bad permissions for mapped region at address 0xFFF0000A0
==29276==    at 0xFFF0000A0: ???
==29276==    by 0x4E5460F: (below main) (in /usr/lib/libc-2.22.so)

内存权限错误，看不太懂。群里问了下，某人回复，栈保护。关掉栈保护再运行就ok了。早晨起床测试了下，果然正常运行。

然后学习了一下栈保护的知识。

首先看看下面这段代码：

#include <stdio.h>

int main(void) {
    char buf[] =
    "\x48\xba\x00\x00\x00\x00\x00\x00\x00\x00"
    "\x48\xbe\x00\x00\x00\x00\x00\x00\x00\x00"
    "\x48\x8d\x3d\x0c\x00\x00\x00"
    "\x48\xb8\x3b\x00\x00\x00\x00\x00\x00\x00"
    "\x0f\x05"
    "/bin/sh";
  void *p = &buf;

  asm("jmp *%0"
      :
      :"r"(p)
      );
}

熟悉缓冲区溢出的同学很明白的就看出这是一段用来获取 shell 权限的 shellcode。运行这段代码，然后就 return 回来一个 shell 了。

怎么抵抗这种缓冲区溢出的攻击呢？这里，我们知道，buf 是局部变量，其运行时内存布局位于栈中。这段代码的问题就在于，buf 本来只是一段数据，最后却被当作代码执行了。我们知道，运行时的内存布局是有各个段的，每个段都有其权限，rwx 分别代表可读可写可执行。上面这个问题，我们就可以通过修改权限避免。将buf 所在的段，也就是栈区设置成不可执行，就没问题了。

事实上，gcc 编译代码的过程中，默认设置的栈区就是不可执行的。这就是栈保护机制。所以上面那段shellcode 实际上运行也会出现段错误，但是如果我们取消栈保护，那么代码就能正确运行了。

注意到高亮那一行，是 stack 区，其权限是 rw_p，没有可执行权限，这个时候栈区的数据就无法作为代码运行了。

然后我们取消该目标文件的栈保护，这里我们使用 execstack 程序，安装该程序：

sudo pacman -S prelink

gcc 怎么做的呢？

  • 使用 gs 验证码机制。当申请一个缓冲区后，在向缓冲区写入数据之前，gcc 会在缓冲区的结束地址之后放入随机的 gs 验证码，然后在写入操作结束之后检查该验证码有没有被覆盖，一旦被覆盖，那么就说明缓冲区已经溢出。为了关闭这个机制，编译时候可以加上 -fno-stack-protector 选项。
  • ld 链接器，ld 链接程序时候，如果所有 .o 文件的堆栈都不可执行，那么整个库的堆栈段才会被标记不可执行。如果有一个 .o 文件被标记为可执行，那么整个库的堆栈段都会可执行。为了关闭 ld 链接器不可执行机制，可以在链接阶段使用 -z execstack 选项关闭。

那么什么情况下，可能需要关闭栈保护呢？在一般非恶意的情况下，一个显著的情况就是使用了嵌套函数（Nested Functions），如果你写出了这样的代码，一般为了安全，你可能需要重写你的程序，重写有时候是简单的，有时候却是复杂的。
