意图和动机

如果有需要保证系统里一个类最多只有一个实例时，我们就需要用到单例模式，比如缓存池、数据库连接池、应用服务实例等等。这个时候，单例保证系统里仅有一个实例，并提供一个访问它的全局访问点。一个全局的变量就可以做到容易被访问，但是要做到一个类只被实例化一个对象，比较好的做法就是让类自己管理自己的实例，然后提供一个访问该实例的方法。

创建私有对象

根据前面的想法，一个比较直接的做法就是，定义一个私有的类实例变量，提供私有的构造方法，私有方法的 private 的修饰符保证了该变量不会被客户类直接访问。同时为了客户类能够得到该实例，我们需要提供一个 get 方法。

public class Singleton {
    Iprivate static Singleton instance = new Singleton();  
    
    private Singleton(){};  //私有构造方法保证不被调用
    
    public static Singleton getInstance() {
        return instance;
    }
}

这里，其实也可以将 instance 属性直接设置成 public，这样，客户类就能直接访问该属性，但是这样，会导致客户类和该类的属性名称耦合，如果改变了该类的该属性名称，会引起客户端的变化。

延迟创建

上面的代码中，在加载类的过程中，其属性值 instance 会被初始化，这个时候，该类就已经实例化了。然而，很多时候，处于性能的考虑，我们可能想延迟实例化。毕竟并不是所有的类都是这么简单的，相反，单例模式的很多类其实比较复杂，消耗占用资源都不少。在不需要使用该类的时候，我们并不想实例化它。这个时候，就需要延迟创建实例了。多说一句，Java 中有很多 lazy 的模式，其实都是出于这方面的考虑，不管是延迟加载还是延迟创建。

要想延迟创建该类，而不是在类初始化时候就创建，那么我们只需要将对构造方法的调用放到 getInstance 中去，同时，对该实例是否已经存在做一个判断即可。

public class Singleton {
    private static Singleton instance;

    public String name = "telnetning";

    public static Singleton  getInstance() {
        if(instance == null) {    //mark1
            //判断实例是否已经存在    
            instance = new Singleton();    //mark2
         }

        return instance;
    }

    private Singleton(){}
}

上面代码中，我们多定义了一个 name 属性，用来对单例模式做个测试。

public class TestSingleton {
    public static void main(String[] args) {
        Singleton s1 = Singleton.getInstance();
        System.out.println(s1.name);
        s1.name = "ningtelnet";
        Singleton s2 = Singleton.getInstance();
        System.out.println(s2.name);
    }
}

我们能够看到输出为：

telnetning
ningtelnet

可以看出，s1 和 s2 确实指向同一个实例。

线程安全问题

在上面延迟创建实例的代码中，注释为 mark1 和 mark2 的两行，可能出现线程安全问题。

当两个 Thread 同时跑到 mark1 时，它们都发现 instance == null 成立，然后，都去执行 mark2，这个时候，多个实例就产生了，这不是我们想要的结果。

在需要手动回收内存的语言中，比如 C++，可能出现更大的问题。我们认为只有一个实例产生了，最终也只手动回收了一个实例，然后就导致了内存泄露。

这个时候，我们需要采取同步策略。在 Java 中，我们使用 synchronized 对 getInstance 方法进行同步。

    public class ThreadSafeSingleton {
        //variable and constructors...
        
        public static synchronized ThreadSafeSingleton getInstance() {
            if(instance == null) {  //mark1
                instance = new ThreadSafeSingleton();  //mark2
            }
            return instance;
        }
    }

这样，多个 Thread 不可能同时执行 getInstance 方法，也就不可能产生多个实例了。

Double-Check Locking

上面对 getInstance 方法进行同步，达到了我们想要的同步效果。但是使用 synchronized 会导致性能的下降。其实我们想要干的只是保证多个线程不会同时执行到 mark1 和 mark2 处。所以我们改为对该代码段进行同步。

    public class DoubleCheckSingleton {
        private volatile static DoubleCheckSingleton instance = null;
        
        //constructor
        
        public static DoubleCheckSingleton getInstance() {
            if(instance == null) {  //mark1
            
                synchronized(DoubleCheckSingleton.class) {
                    if(instance == null) {
                        instance = new DoubleCheckSingleton();
                    }
                }
                
            }
            
            return instance;
        }
    }

这里使用了两次 instance == null 的判断。原因是，同步能保证两个线程不同时执行，但是它们依旧都会被执行。如果两个线程同时到达 mark1 处，它们都进行了 if 判断体内。这个时候，不管哪一个线程先占据同步锁，并创建实例，也没法阻止另一个线程在该线程释放锁之后进入同步块，创建实例。因为，另一个线程，此时也在 if 判断体内了。因此，我们需要在同步块内再次判断 instance 是否为 null。

因此，这种方式叫做，Double-Check Locking。

另外，属性 instance 被 volatile 修饰，使得该属性的变化能立即别其它线程发现。

需要注意的是，在 Java5 之前，由于 out-of-order writes 的原因，该代码可能执行失败。此时，JVM 允许实例尚未完全实例化时，instance 变量就指向该实例。这个时候，其它线程发现 instance 不为 null，然后引用 instance，可能造成程序崩溃。

Initialization on demand holder

这一种单例模式的实现方式，拥有单例、延迟创建以及线程安全的特性。它利用内部静态类来实现。

public class LazyLoadedSingleton {
    private LazyLoadedSingleton(){}

    private static class LazyHolder {
        private static final LazyLoadedSingleton singletonInstance = new LazyLoadedSingleton();
    }

    public static LazyLoadedSingleton getInstance() {
        return LazyHolder.singletonInstance;
    }
}

当该类被加载的时候，由于没有 static 代码，因此并不会有什么特殊操作。当第一次调用 getInstance 方法的时候，内部静态类 LazyHolder 会被加载，创建外部单例类的实例，而在类的加载过程中，Static 属性以及 Static 初始化块的执行时串行的，这就保证了不会同时 new LazyLoadedSingleton() 有两个线程在执行。

这一种方式也被称作 SingletonHolder。

Singleton 的序列化

根据以上的方式，似乎就可以完全实现单例模式。然而，我们却疏忽了，类的实例不止是可以由构造函数产生。使用反序列化，我们也可以直接生成一个类的实例。如果一个单例类已经存在，我们利用反序列化又生成了它的一个实例，这个时候单例就不成立了。

好在，Java 的序列化接口，给我们提供了一个方法叫做readResolve，让我们在反序列化之前做一些操作。

readResolve 方法允许在把从流（stream）中读入的对象（Object）返回给调用者（Caller）之前，对该对象进行替换（Replace/Resolve）。

比如我们要从 ObjectInputStream 中读入一个对象。在从 ObjectInputStream 读入了该对象之后，在将该对象返回给调用者之前，会检查该类的 readResolve 是否有定义，如果有定义，则会将调用 readResolve，readResolve 方法就可以替换掉我们从 ObjectInputStream 中读入的对象。

这样，即使是序列化，也是直接返回我们在内存中保存的那个唯一的单例对象。

其它

在 Spring 中定义 Bean 的时候，可以通过 scope="singleton" 将 bean 定义成单例模式。单例模式也是 Bean 的默认行为方式。

Spring 推荐将所有的业务逻辑组件、DAO 组件、数据源等配置成单例模式，因为这些组件无需保存任何用户状态，故所有客户端都可以共享这些组件。

使用单例模式的优势：

  • 减少 Java 创建实例的开销
  • 便于系统跟踪单个实例的生命周期、实例状态等
