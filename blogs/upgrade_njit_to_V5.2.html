升级 njit-client 到 V5.2

0x00

以前学校（西电）的客户端是 V3.6 版本的，当时为了简单，我直接买的一个刷好了 OpenWrt 的 Huawei Hg255 的路由器。去年下半年学校升级了 H3C 的客户端，我的路由器不能使了。运行 njit-client 时候会提示：

the client is too old

查看了一下 windows 上的 iNode 客户端，版本显示为 V5.2。那么我需要升级 OpenWrt 中的 njit-client 了。
要想升级 njit-client，考虑有以下几种方法：

  • 交叉编译 njit-client，就是在 Linux 上编译 OpenWrt 的工具链，然后使用该工具链来编译 njit-client 源码。
  • 将 njit-client 与 OpenWrt 一起编译，当时不编译进 OpenWrt 中，只是编译成模块 。
  • 将 njit-client 直接编译进 OpenWrt 中去，然后重刷路由器中的 OpenWrt。

第一种方法和第二种方法都不需要重刷 OpenWrt，所以成了我首先想到的方法。
第一种方法，交叉编译，参照 交叉编译 openwrt 平台上的开源 H3C iNode 客户端 njit-client，然后失败，在编译完工具链之后编译 njit-client 的时候报错大致如下：

In file included from /usr/include/features.h:356:0
    from /usr/include/stdio.h:28
    form /home/telnetning/development/openwrt/openwrt/staging_dir/toolchain－mipsel_24kec+dsp_gcc-4.8-linaro_musl-1.1.10/include/fortity/stdio.h:20

明显是交叉编译的环境配置有问题，但是我对交叉编译也没那么熟悉，一时懒惰，没有 debug。
换第二种方式，同样出现了不可思议的问题，一切进行地 ok，生成的 ipk 文件在 OpenWrt 上缺无法安装，报错为不兼容的架构，虽然我明明选择了 ramips 架构，目标机器 hg255，实在难以理解。于是换了第三种方法，成功了,下面是我记录的整个过程。

0x01

首先需要更改 njit-client 版本，在 github 上找到 njit-client 的代码，把 master 分支 git clone 下来:

git clone https://github.com/liuqun/njit8021xclient.git

查找关于认证的代码，在 src/auth.c 中，其中第 33-34 行

const char H3C_VERSION[16]="CH V3.60-6303"; // 华为客户端版本号
const char H3C_KEY[64]    ="HuaWei3COM1X";  // H3C的固定密钥

查看 windows 的版本号为 5.2 E0408，于是这里将 33 行更改成：

const char H3c_VERSION[16]="CH V5.20-0408";

第 34 行为固定密钥，由于不知道 V5.2 的密钥，只能依旧写这个，祈求密钥没有变化（因为从 V2.4 到 V3.6 密钥一直也是这个）。后面结果证实密钥确实没有变化。

auth.c 中的 262 行也需要更改，源码为：

system("njit-RefreshIP");

这行代码是在认证成功之后用来获取 ip 的，其中的 njit-client 是一个 python 脚本，我不知道从哪儿引入的，有人将其改成 dhclient，然后我在路由器的 OpenWrt 中野没有发现 dhclient，查阅了资料，知道 OpenWrt 中使用的是 udhcpc 来做 dhcp 客户端，于是，我改成了：

system("udhcpc")

事实是这段代码依旧导致了问题，我发现在认证完成之前端口已经获得了 ip，认证完成后就已经能联网，而一旦再运行到这行，又会导致断网。所以这儿其实只要注释掉这一行就行了（如果你跟我一样在西安电子搞基大学）。

0x02

OpenWrt 的编译过程中会根据 openwrt/package/ 目录来编译软件。比如 package 下面有个 njit8021xclient 目录，这个目录下会有 Makefile 文件，也可能还有 patch 文件夹。编译时首先会到 dl 目录下找有没有该软件包，没有的话就回从网上下载。这里我从 sourceforge 上的 njit-client 主页 下载 Makefile 包，解压得到 njit8021xclient 之后直接放入 package 文件夹。

然后我们需要把自己更改过的 njit-client 源码包伪装成下载的包，将 njit 源码包重新打包:

tar zcvf njit8021xclient-1.3.tar.gz njit8021xclient-1.3/

然后移动该包到 dl 目录下，再计算其 md5 值，

md5sum njit8021xclient-1.3.tar.gz

更改 package/njit8021xclient/Makefile 中的 md5 值为刚刚计算得到的 md5 值：

#PKG_MD5SUM:=1b19ac98ba90e18aff7aeefa431753e2
PKG_MD5SUM:=9fca6b3e228f21b4998dba743f6d702e

在进行make之前我们需要：

sudo mv /usr/bin/pcap-config /usr/bin/pcap-config.bak

根据 njit-client 的作者 liuqun 说法是，pcap 会导致头文件污染。
在编译之前，另外一个工作就是加入 LuCI 到 OpenWrt 中去，

./script/feeds update packages luci
./script/feeds install -a -p luci

然后是一个不知道对结果有没有影响的操作，

vim target/linux/ramips/image/Makefile

打开对 HG255 的支持，把该文件中所有存在 hg255 的行的前面的注释去掉。

0x03

运行 sudo make menuconfig

  • Target System 选择路由器对应的架构，这个在 OpenWrt 的官网上可以查到。
  • Target Profile 找到路由器的型号。
  • LuCI->Collection 下选择 luci 以及 lusi-ssl
  • Network 下选择 njit8021xclient

所有的选择都是 *，而不是 M，表示不是编译成模块，而是直接编译进 OpenWrt 固件中。
然后就可以 sudo make V=99 -j8 了，-j 后面表示编译过程中使用的线程，这个根据机器性能来写。
编译完成后在 bin 目录下有生成的文件，hg255 对应的生成的就是：

openwrt-ramips-rt305x-hg255d-squashfs-sysupgrade.bin

下面刷进路由器，由于之前已经有 OpenWrt，所以直接使用 Web 的方式就行了。刷完设置之后输入：

njit [学号][密码][网卡]

应该就可以了。
