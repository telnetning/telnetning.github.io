Linux内核课堂笔记（5）

李龙海老师的 linux 内核课程第五课的课堂笔记。主要内容是 C 程序运行时内存分布的栈段。

我们首先来看一段在栈上动态分配内存的代码熟悉一下：

void *_alloc( size_t size );

#include <malloc.h> 
void f()
{
    char *p;
    p = (char*) _alloc(20);
    memcpy(p, "Hello, World!", 13);
    printf("%s\n", p);
}

int main()
{
    f();
    return 0;
}

在 main 主函数中我们调用了 f 子函数，在 f 子函数中我们定义了局部变量 p，这是一个指针，然后我们给这个指针动态分配了一块内存。那么我们的问题来了，变量 p 和其指向的内存都是在内存的哪一段呢？

根据前面的知识，我们知道，p 作为一个局部变量，它应该是在栈段的，指针和普通的变量并没有什么本质上的区别的。动态分配的内存呢，则是在堆区的，这个在后面堆区的讲解中我们会强调的。由于其分配的内存在堆区，而且没有释放，这里 f 函数会发生内存泄漏。

还有一个值得注意的就是“Hello, World!”字符串。这里我们比较显示地将该字符串 copy 到了 p 指向的内存区域中，而该字符串本身是存储在常量数据区的。

在子函数中返回指针值的操作很容易出现问题，不信我们来看看几段代码。

首先是一段之前看过的代码：

#include <stdio.h>
#include <stdlib.h>

void getmemory(char *p)
{
    p = (char *)malloc(100);
    strcpy(p, "Hello world");
} 

int main()
{
    char *str = NULL;
    getmemory(str);
    printf("%s\n", str);
    free(str);
    return 0;
}

这段代码我们前面讲过了，肯定是有问题的。原因在于主函数中的 str 实参和形参并不是同一个实体。所以经过子函数的调用，主函数中的 str 依旧指向 NULL。子函数并没有将 p 的值传回给 str，也就是子函数中分配了一块内存，但是没有返回内存的地址给主函数。

好的，要返回地址，我们再看看下面的代码。

#include <stdio.h>

char *GetMemory()
{
    char p[] = "Hello, Wordl!\n";
    return p;
}

int main()
{
    char *str = GetMemory();
    printf(str);
    return 0;
}

这里我们在子函数中把 p 的地址返回了，一切好像都没有问题。仔细想想，事情不是这么简单的。

结果还是不对的，原因呢？老问题，这里在子函数中 p 是啥？是数组。也就是p是数组的首地址。但是这个数组在哪里？这个数组是一个局部变量，它在栈中。所以一旦子函数 ret 了，p 这个数组也就不存在了。实际情况就是，str 依旧指向了 p 的地址，但是 p 段内存里面是什么东西，已经说不清了。

我们对这段程序稍微改那么一下：

#include <stdio.h>

char *GetMomory()
{
    char *p = "Hello, World!";
    return p;
}

int main()
{
    char *str = GetMemory();
    printf(str);
    return 0;
}

这段程序和上面那段程序唯一的区别就是把p定义成了一个指针，而不是直接一个数组。但是这段代码没有任何问题了。

分析一下，p 是一个指针，它指向一个字符串，这个字符串在哪里？是不是在栈中？不是！它是一个字符串常量，它在常量数据区。所以 p 的值是常量数据区的地址。当我们从子函数中 ret 回来，把 p 的值给了 str，那么 str 指向的也是这个地址。由于常量数据区不是栈区，子函数结束后，这段地址的访问依旧有效。所以现在 str 就是指向常量数据区的"Hello, world!"常量字符串的指针。我们现在打印这个字符串，就没有任何的问题。

从上面的这几段代码，我们能够感觉到，很多时候，对运行时内存动态分布的理解，常常决定了对代码的直接理解能力。程序中的很多问题，都是因为对内存分布理解不够造成的。栈区和后面要讲的堆区时最容易出错的地方。

同时，通过上面的两段程序，也可以理解一下指针和数组是不是完全一样的。

接下来我们再来着重看一下栈在函数返回值中的作用。
首先我们用点时间看一下在 C 中如何嵌入汇编编程。我们在这段程序中关注一下：

  • 汇编如何使用寄存器
  • 汇编中如何访问C代码中的变量
  • 汇编中如何调用C代码中的函数
#include <stdio.h>

int g = 0;
void f()(int x, int y)
{
    printf("x=%d, y=%d\n", x , y);
}

int main()
{
    int a, b;
    __asm {
        mov dword ptr [g], 3
        mov dword ptr [a], 5
        mov dword ptr [ebp-8], 5
        push 7
        push 6
        call f
        add esp, 8
    }
    printf("g=%d, a=%d, b=%d\n", g, a, b);
    return 0;
}

我们来分析下汇编代码：

  • 第 13 行，这里将 3 赋值给 g，加上中括号表示取其地址之意。
  • 第 14 行，将 5 赋值给 a。
  • 第 15 行，将 5 赋值给某个地址，这个地址是什么呢，ebp 是栈帧底部。ebp－8 的位置，就是栈帧增长8的位置。是啥？我们看看栈里有啥，局部变量 a 和 b，a 的位置就是 ebp－4，ebp－8 呢，就是 b 啦。
  • 第 16 行，把 7 压栈。
  • 第 17 行，把 6 压栈。这两句啥意思？就是给实参压栈。
  • 第 18 行，调用 f 函数。
  • 第 19 行，esp 加 8，就是退栈，释放了 8 个字节的空间，猜猜是啥？就是子函数中的两个形参 x 和 y。

这里我们可以看看形参数和实参。运行代码后，得到的结果是 x＝6，y＝7，说明先压栈的 7 被较右的形参获得。这是一个形参和实参的对应情况。

另外，这段代码时 Intel 汇编，只能在 Windows 上跑。Linux 下 GCC 用的是 AT&T 汇编。很多人说，GCC 也可以设置支持 Intel 汇编啊，但是 GCC 中使用 inline 的 Intel 汇编，它居然还不支持局部变量以及函数参数。

这个例子中子函数没有返回值，但是想一想，如果一个函数有返回值，返回值是怎么返回给主函数的呢？毕竟说了这么久的栈，我们好像并没有在栈中看到任何空间是用于存储函数的返回值的。

结论直接说明：

  • 如果返回值小于等于 4 个字节，那么用 EAX 寄存器返回。
  • 如果返回值介于 4 到 8 个字节，那么用 EAX 和 EDX 寄存器返回。

怎么验证上面的结论呢？我们来看一段代码：

#include <stdio.h>
int get_sp(){
    __asm__("movl $0x08, %eax");
}

void main(){
    printf("%x\n", get_sp())
}

用 gcc 编译运行下，你会得到结果 8。怎么回事，我们并没有在子函数中返回任何数啊。原因在哪里？我们在子函数中给 eax 置了个值 8。于是子函数返回的时候，主函数从 eax 中读取子函数的返回值，就读到了 8。

我们稍微改一下呢：

#include <stdio.h>
unsigned long get_sp(){
    __asm__("movl $0x08, %eax");
}

void main(){
    printf("%x\n%x\n", get_sp())
}

我们得到的第一个结果是 8，第二个结果不知道是什么，好像是一串随机的数。这串随机数是什么？这里先不说，我们将代码再改一下：

#include <stdio.h>
unsigned long get_sp(){
    __asm__("movl $0x08, %eax");
    __asm__("movl $0x09, %edx");
}

void main(){
    printf("%x\n", get_sp())
}

得到的结果是打印了 8 和 9。注意一下子函数的返回值的长度。这个时候能猜到上面那个程序打印的随机数是什么了么？还是不知道的话，可以把这里的 edx 换成 ecx 之类的，看看结果。你应该能知道的。

简而言之，就是用寄存器存储返回的值，一个寄存器够，就用 EAX，两个才够，就用 EAX 和 EDX。两个都不够呢？也就是大于 8 个字节的时候怎么办？

是这样的：

  • 在调用者的栈帧中开辟一块临时空间，其大小刚好等于返回值对象。（生成一个临时对象）
  • 将临时对象地址作为一个隐含的参数通过栈传递给函数
  • 函数将运算结果拷贝到临时对象，并将临时对象地址用 EAX 传递给调用者
  • 调用者再将临时对象拷贝给非临时对象

举个例子，main 函数调用子函数，返回是一个 1000 个元素的数组，那么在 main 函数的栈帧中，会开辟一个临时的空间，恰好为 1000 个字节，然后这个空间的地址会被传到子函数中，子函数在返回前，把应该返回的结果 copy 到该临时空间中。main 函数中再把临时空间的内容 copy 给非临时变量。

struct A{ int a［1000］;};

A f(){
    A t;
    return t;
}

main(){
    A x;
    x = f();
}

这里 x 是实参，t 是子函数中的局部变量。要把 t 的内容返回回来，给 x。在这儿使用了临时变量。

t -> tmp -> x

tmp 作为一个临时空间，在函数返回的过程中担任了桥梁的作用。

还有一点比较重要的内容，也和栈有一定关系，就是线程的安全性和函数的可重入性。

这个话题说来重要且话长，但是这里只能粗略说说。

进程是资源分配的单位，线程时执行的单位。线程共享进程的内存空间，但是线程有自己的栈空间。

如果一个函数（子函数）在多线程环境下与预定义的行为一致，称其满足线程安全性。

举个例子，多个线程对共享内存中的一个链表进行插入操作。这个就是线程不安全的，很明显，容易产生冲突。怎么办呢？只有进去函数后对资源加锁，然后操作完成后再进行解锁。但是这样的多线程在这个操作上来说，其实没有什么意义了，在这里，它的执行变成了单线程。

在并发环境下，如果一个函数没有执行完，由于外部因素，又一次进入该函数执行，那么称为重入执行。什么叫可重入函数呢？意思就是，这个函数只依赖自己栈上得变量，不依赖任何外在的全局变量等等，这样的结果就是，多个该函数的副本可以并行运行，而不会相互干扰，不会产生任何副作用。

可重入函数在并行环境下的意义重大。有一种思想，所有的问题都可以由一系列不含副作用的函数来解决，这个被证实与图灵机模型等价。这几年并行多线程多核越来越多，也就越来越需要这种思想，这里的不含副作用的函数，就是可重入的函数。而想 scala 等函数式语言就是这种思想，一切靠可重入的函数解决。所以，函数式语言变得越来越流行。
