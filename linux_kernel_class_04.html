Linux内核课堂笔记（4）

李龙海老师的 linux 内核课程的第四课的听课笔记。涉及内容：运行时栈，缓冲区溢出攻击基本原理。

在前面已经讲到过，在程序运行的过程中，内存布局中有栈这一个段。每一个进程独享 4G 的内存地址，用户空间使用 3G 以内的地址，栈在这 3G 内存的高位，从上向下增长。这节课的主要内容就是系统栈以及过程调用。

一般只要不是初学者，写程序，都是 main 函数调用 A 函数，A 函数再调用 B 函数这个样子。我们看看这个过程中的栈的使用情况。

首先看看下面这段代码：

int func_B(int arg_B1, int arg_B2)
{
    int var_B1;
    int var_B2;
    return 0;
}

int func_A(int arg_A1, int arg_A2)
{
    int var_A1;
    int var_A2;
    func_B(4, 3);
    return 0;
}

int main(){
    int var_main;
    func_A(2, 1)
}

这里我们按照程序运行的顺序，一步步来看栈的状态。

  • 首先是第 17 行，申明了一个局部变量，这个变量会被压栈。
  • 接下来第 18 行，调用 func_A，注意，这里有两个实参，分别为 2 和 1，这里在调用子函数之前，会有很多事发生。首先系统需要为两个实参分配内存，并且赋值，所以，将参数值1压栈，再将参数值 2 压栈。这里的顺序值得关注，它是从右往左的顺序对参数进行压栈的。
  • 还是第 18 行，对两个实参压栈之后，要准备调用子函数 func_A 了，在进入 A 之前，还有一个十分重要的事情要做。就是将调用之前，将当前指令的下一条指令压栈（这里的下一条指令就是编译器给 main 函数加上的 return）。然后进入子函数 func_A，在这期间，编译器还会存储一些其它信息。
  • 进入 func_A 函数，同样的，首先对两个局部变量压栈，然后准备调用 func_B，在调用之前，对两个实参压栈，把 return 地址压栈。
  • 进入 func_B 函数，压栈两个局部变量 var_B1, var_B2。

上面的"其它信息"指是什么呢？下面会看到。

总体的栈的情况如下：

这里我们先说一下栈帧的概念。在图中我们可以看到，这里我们可以把栈分为 main 的栈帧，func_A 的栈帧，func_B 的栈帧。什么意思呢，各个函数在栈中的信息我们看作一个整体，这个就叫做一个栈帧。栈帧有什么作用呢？这个跟EBP这个寄存器密不可分。首先我们看看 CPU 对过程调用的支持这些比较基础的概念。

CPU 对过程调用的支持相关的寄存器主要有三个，ESP，EBP 以及 EIP。

  • 首先说说 ESP，对于一个栈，从栈底往栈顶增长，那么肯定得记录栈顶增长到哪儿了，这个就是寄存器 ESP 的作用。ESP 寄存器存放一个指针，它始终指向系统栈的最上面一个帧栈的栈顶，也就是整个系统栈的栈顶。
  • 有指向栈顶的，那么有指向栈底的么？有，EBP 就是。但是呢，EBP 并不是指向整个系统栈的栈底，它指向的是系统栈最上面那个栈帧的栈底，也就是当前栈的栈底。有时候也被称为栈帧寄存器。什么作用呢？后面会讲到。
  • EIP 这个，同样存放一个指针，指向下一条等待执行的指令地址。

我们说说上面这个 EIP 寄存器。它指向的是下一条指令的地址。在上面的栈结构中，我们曾经在进入函数之前，将调用函数语句的下一语句压栈，比如上面的第 12 行调用 func_B，在进入 func_B 之前，第 13 行的 return 0;语句的地址被压栈了。这样的效果是什么呢，就是从子函数返回的时候，我们从栈中取出这个存储的返回地址，就能知道下一条语句应该执行什么。

函数调用的机器指令为 call，那么我们根据上面的内容我们应该都能猜到 call 的内容了。call 会将当前语句（也就是调用子函数的这条语句）的下一条语句的地址（也就是此时 EIP 的值）压栈，然后跳转到子函数。实际上就是如此。我们看看与 CPU 提供的与过程调用的机器指令：


我们来分析下：

  • push 一个操作数，也就是将这个数压栈，所以栈顶会加 1（并不是 1 个字节的意思，实际上是 4 个字节）,但是栈是从内存的高端往低端增长，所以效果就是指向栈顶的 ESP 指针减 1，然后将操作数的值填到 EDP 指向的栈顶。
  • pop 操作，与 push 相反，首先返回 ESP 指向的值，然后出栈，也就是 ESP 加 1。
  • call 操作，上面说过了。call 和 jmp 的差别就是 call 会压栈下一条指令的地址。
  • ret 操作。函数的返回操作，当要返回时候，只要把返回地址取回来放到 EIP 寄存器中就行了。栈的管理使得 ret 操作时候从栈中取出来的地址是应该返回的地址，但是返回地址有可能被恶意改变。这个我们后面会看到。

根据上面的知识，我们对一段 C 程序来进行简单的汇编分析。

void f(int x, int y)
{
    int a, b;
    a = 7, b=9;
    x = x + y;
}

int g = 3;
int main()
{
    f(g, 5);
    return 0;
}

看看汇编，

//main函数对应的汇编
push    5   //将实参5入栈
mov     eax,[00422d8c]; g = [00422d8c]  //将变量g的值存到eax寄存器中
push    eas //将实参g的值入栈
call    f   //调用子函数
add     esp,8   

//子函数对应的汇编
push    ebp //保存上层函数（main函数）的栈帧
mov     ebp,esp //保存当前栈的栈底
sub     esp,8   //为两个局部变量分配空间
mov     [ebp-4h],7  //a = 7
mov     [ebp-8h],9  //b = 9  
mov     eax,[ebp+8h]    //实参3(x)存到寄存器中
add     eax,[ebp+0ch]   //实参5(y)和实参5相加
mov     [ebp+8],eax     //x = x + y
mov     esp,ebp     //回收局部变量空间
pop     ebp     //从栈中取出上层函数的栈底地址，将ebp指向上层函数的栈底
ret         //从栈中取出返回地址，置EIP，下一条指令就返回上层函数

这里可以比较清楚的看出来过程调用中的栈的情况。可以注意到：

  • 进入子函数前保存现场，是通过压栈上层函数的栈底地址以及下一条指令的地址来实现的，它们保证了函数调用返回的时候，上层函数的栈帧能够重新恢复，并且找到下一条应该运行的语句。
  • 全局参数的地址可以通过唯一内存地址得到。那么局部参数的地址呢？由于局部变量的地址都是运行时生成的，无法直接使用其地址。在上面我们看到，局部变量的地址都是通过相对EBP定位得到的。并且，函数的参数都在EBP所指示的内存地址的正偏移处，函数内的局部变量都在EBP所指示内存地址的负偏移处。

这张图比较重要。我们好好看看这张图，然后思考下，如果缓冲区溢出，覆盖了别的变量，甚至返回地址，会出现什么情况？这就是缓冲区溢出的基本原理。

我们先来看一看下面这段代码：

#include <stdio.h>
voif f()
{
    int i, a[10];
    for(i=0; i<=10; i++){
        a[i] = 3;
    }
}

int main()
{
    f();
    printf("main() function\n");
    return 0;
}

输出是什么呢？仔细看看，在 f 函数中，我们越界访问了数组，结果是什么呢？

首先，编译器是不会报错的，C/C++ 的编译器不做越界检查。那么运行呢？会报内存错误么？
实际上，不会，会陷入死循环。为什么？我们画画内存图：

-------
  ...
-------
  i
-------
 a[9]
-------
 a[8]
-------
  ...
-------
 a[1]
-------
 a[0]
-------

我们可以顺便关注一下数组在内存中的排列。不管在什么机器中，数组的第一个元素永远是地址最小的。

在代码中，我们访问了 a[10]，在 i=10 的时候。本来接下来 i=11，循环就应该终止了。但是呢，我们让 a[10]=3，实际上呢，让 a[9] 上面的元素，值为 3 了，这个内存是什么？ i！所以在 i=10 的时候，我们将 i 又变成 3 了，所以，循环会继续跑，周而复始，陷入死循环了。

然而，有些时候，缓冲区溢出，造成的后果可不止这样。我们再来看看下面这段代码：

#define PASSWORD "1234567"
int verify_password(char *password)
{
    int valid_flag = 0;
    char buffer[8];
    valid_flag = strcmp(password, PASSWORD);
    strcpy(buffer, password);
    return valid_flag;
}

void main()
{
    char inputstr[1024];
    for(int i=0; i<3; i++){
        printf("Please input password:"); 
        scanf("%s", inputstr);
        if(verify_password(inputstr)==0){
            printf("You have passed the verification!\n\n");
            /*do something else*/ 
        } else {
            printf("Incorrect password!\n\n"); 
        }
    }
}

这是一段密码验证程序。在主函数中，有三次密码输入的机会，输入之后的字符串会和真实的密码作比较。一旦匹配，认证成功，否则，认证失败。

怎么样能绕过这段密码验证呢？

还是先开画一下内存：

------------
 valid_flag
------------
 buff[7-4]
------------
 buff[3-0]
------------

根据上面的经验，我们需要覆盖掉 valid_flag，而 valid_flag 为 0 的时候，代表验证通过，也就是我们要将其覆盖成 0。考虑一下 valid_flag 的内存，一共是 32 个 bit，也就是 4 个字节。

|32-25bit|25-17bit|16-9bit|8-1bit|

这里由于缓冲区 buff 是 char 类型，所以我们一次只能覆盖 valid_flag 的一个字节。能否将其 4 个字节全部覆盖呢？也就是全部覆盖成 0 。不能的。这里我们利用的是 strcpy 这个漏洞，越界访问 buff 来覆盖 valid_flag。

我们需要注意的是，strcpy 的原理是，遇到""就停止复制。而如果我们要把 valid_flag 的4个字节全部覆盖的话，我们需要输入"xxxxxxxx"，这样才能保证 valid_flag 的所有位为 0，也就是值为 0。但是，两个问题：

  • 我们如何用键盘输入""这个字符？
  • 由于我们利用的是 strcpy 漏洞，即使输入如上，strcpy 复制到 buff 中的字符串，也只是"xxxxxxxx"，而不会有后面的三个""。

所以，怎么办？我们可以利用字符串的""结尾的原理，覆盖 valid_flag 的最低位字节为全 0，然后使用其它方式，让其高位为 0。怎么让其高位为 0，看第六行的 strcmp 这个字符串比较函数。

strcmp 的返回值有 -1,0,1 三种，0 表示两个字符串匹配，1 表示前面一个字典序大，-1 表示后面一个字典序大。

valid_flag 作为其返回值，我们要将其高 24bit 全部置为 0，那么 strcmp 的结果只能为 0 或者 1，为 0 不现实，不然我们费劲在干嘛。所以我们需要让其返回值为 1。也就是让我们输入的字符串的字典序比密码的字典序大。这个很好实现。

然后我们输入 8 位字符。

结果就是，我们输入的字符串在 strcpy 的时候，会将""复制到 valid_flag 的低 8bit 去。结合 strcmp 的结果，我们就能绕过密码验证了。

这就是缓冲区溢出的危害。

最后，我们再来看看一段“飞了”的程序：

#include <stdio.h>

void g()
{
    printf("g() function\n");
}

void f()
{
    int a[10];
    a[11] = (int)g;
}

int main()
{
    f();
    printf("main() function\n");
    return 0;
}

这段代码的原意是调用一下 f 函数，并无意调用 g 函数。但是在 f 函数中，明显越界访问了。

a[10] 其实就已经越界了，为什么改变了 a[11] 这个元素呢？因为这是精心构造过的。

我们回顾一下上面的栈结构图就知道了。a数组上面的紧挨着的，是上层函数EBP的地址，再上面是返回地址。它们可以分别用 a[10] 以及 a[11] 来覆盖。

改变 a[11] 的意思是啥呢？覆盖了返回地址，也就是子函数返回到主函数后运行的下一条语句。这个地址被改变到另外的函数 g，也就是本来应该返回到主函数的程序，现在返回到了 g 函数里，然后 g 函数开始运行。想一想，g 函数如果是黑客自己写的一段代码……

这就是缓冲区攻击的另外的危害。
