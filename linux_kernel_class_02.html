Linux 内核课堂笔记（2）

李龙海老师的 linux 内核课程的第二课的听课笔记。涉及内容：结构体的内存表示;如何使用 C 实现一套通用的链表函数库;联合体的特殊作用;基本的内存映射知识。

结构体的内存表示

#include <stdio.h>
typedef struct _A {
    int a;
    char b;
    float c;
}
 void main()
 {
     printf("%d\n", sizeof(A));
     return;
 }

这样的一段程序，求一个结构体的长度，答案是多少呢？
一个对C语言的内存了解不多的人，比较容易回答是 9，原因是：

4(int) + 1(char) +4(float) = 9

有人会说了，这个不对，在不同机器上不一样的。确实，比如在 16、32 以及 64 位机器上 int 型的长度就不一样。

那我们将机器类型定为 32 位的呢？这个题目在笔试题目中经常会出现，考察的其实就是内存对齐策略。一般懂的人会说，根据内存对齐，应该是 12 个字节。

12 个字节对么？其实这个回答也只是对了 95%，更准确的回答是什么呢？不一定。

为什么不一定呢，原因是，内存对齐的策略是可以改变的。比如在 gcc 中，我们可以这样做：

struct A{
  ...  
}__attribute__((packed));

什么意思呢，__attribute__ 是 GNU C 中的一个特色，使用它可以更改设置函数属性、变量属性以及类型属性。

这里就是利用 __attribute__ 设置结构体的属性，packed 表示取消对齐优化。

而

struct B{
  ...
}__attribute__((aligned(8)));

可以用来设置对齐方式，这里的 8 也可以取成 1、2、4。

同时编译器中也提供了以下选项来设置对齐方式：

#pragma pack(n)
#pragma pack()

分别用来设置对齐以及取消对齐设置。

C中通用链表函数库的设计

在 C 中，使用链表的地方非常多，比如，我们来设计一个 fox 的链表，首先我们定义一个 fox 的结构：

struct fox {
    unsigned long tail_length;
    unsigned long weight;
    bool          is_fantastic;
}

然后为了能够形成链表，我们在结构体中加上指针：

struct fox {
    unsigned long tail_length;
    unsigned long weight;
    bool          is_fantastic;
    struct        *next;
    struct        *prev;
}

然后我们就会写一系列的函数，增删改查等这些，比如一个 add 函数，像链表中添加一个元素，它的函数定义大概是这个样子的：

int add(struct fox *table, struct fox *node)

意思是向 table 表中添加一个 node 节点。注意下，这里的参数都是指向 fox 结构体的指针。

这里一切看起来都是不错的，只是，如果现在我们又需要一个表，比如一群 dog，我们要给它们建链表，怎么办？

由于 dog 的结构类型和 fox 的结构类型是不一样的，我们要怎么办？将上面的步骤全部再来一遍？所有的增删改查函数再写一遍？

再如果，我们要将 dog 和 fox 扔到一个链表上呢？有可能么？它们都不是一样的结构体啊。

linux 中有各种各样的链表，linux 中的链表操作是怎么写的呢？难道针对每种 struct 写一套函数？

当然不是。我们看看 linux 中是怎么做的。

首先，定义一个小结构体：

struct list_head {
    struct list_head *next;
    struct list_head *prev;
}

很明显，这个小结构体可以用来生成一个双向的链表。

然后，重点来了：

struct fox {
    unsigned long     tail_length;
    unsigned long     weight;
    bool              is_fantastic;
    struct list_head  list;
}

这里我们将 list_head 这个结构体作为 fox 的一个子元素。

然后所有对 fox 链表的操作转换成对 list_head 链表的操作，比如：

list_add(struct list_head *new, struct list_head *head);
list_add_tail(struct list_head *new, struct list_head *head);
list_del(struct list_head *entry);

这个时候所有的操作的参数都是 list_head 类型的。而并不用管 list_head 这个之外的内存是什么类型的。

如上图所示，由于我们的这种定义操作，我们甚至可以将不同的结构提法维护在一个链表中。

比如 list_add 的操作：

加入一个 fox 节点：

list_add(&fox->list, &table);

我们也可以加入一个 dog 节点：

list_add(&dog->list, &table);

这样的一套函数库其实是可以用于任意的链表的，只要你在你的链表中包含 list_head 类型的小结构体。

然而，这样使用的时候我们还需要考虑一个问题，如何访问大结构体中的元素呢，比如在图中，我们要如何访问 fox 类型节点的 weight 和 tail 元素呢？

一般访问结构体的使用中，比如我们有指向结构体的指针 ptr，我们能这么访问：

ptr->weight;
ptr->tail

其实 ptr 就是指向结构体的起始地址，其实我们也可以直接使用地址来访问，比如在 fox 的 struct 的定义中（list_head 在最后面的那个定义），weight 的地址就是 ptr+4,所以我们也可以用 *(ptr+4) 来访问。

所以这里我们要想访问链表中的节点中的元素，就得根据 list_head 的地址来计算其它元素的地址。在其前面的的元素地址减特定值，在其后面的元素地址加特定值。 linux 中是依靠宏定义来完成这一过程的。具体定义如下：

#define container_of(ptr, type, member) ({                  \
    const typeof( ((type*)0)->member ) *__mptr = (ptr);     \
    (type *)( (char *)__mptr - offsetof(type, member));     \
})
#define list_entry(ptr, type, member)   \
container_of(ptr, type, member)

#define offsetof(s ,m)(size_t)&(((s*)0)->m)

调用的时候，这么用：

struct list_head *p;
struct fox *f;
f = list_entry(p, struct fox, list);

也就是根据 list_head 的地址得到大的结构体的起始地址。

在 container_of 的定义中：

  • 第二行的意思是将list_head在结构体的地址赋给 __mptr，之所以要这么中转一下，可能 是相当于对 ptr 做了一个类型检查，如果 ptr 不是 member 类型，这条语句编译不会通过。
  • 第三行之所以要用（char *）强制类型转换一下，是因为不同类型的指针减去不同的数减去的字节数是不一样的，比如 int * 型的减 1 相当于减 4，这里的计算内存地址单位是字节，所以必须将指针转换成 char * 类型去减相对偏移才能得到正确的起始地址。

得到大结构体的起始地址之后，我们就可以很容易地访问结构体的各个元素。

在 Linux 中，我们知道，所有的进程都是 init 的后代进程，这些进程构成了一棵进程树。这些进程树使用类似结构体的方式存储,然后各个节点的内容不是一样的，然而仍然能形成一棵树，就是使用类似上面的这种技巧。

联合体以及位域

C 中的联合体和结构体不同地方在于，联合体的元素共享内存。

由于联合体的这个特性，联合体有一些特殊的妙用。

  1. 进行超级强制类型转换

在某些时候，如果你要使用强制类型转换，可能会转换不成功。而使用超级强制类型转换，就不存在转换不成功的问题。

struct test {
    union {
        char a;
        int b;
    }
}

这个时候当实例化一个 test 结构体的时候，test.a 和 test.b 是共享同一块存储的，所以，对它们的改变都会体现在另一个上。只需要根据需要访问 test.a 或者 test.b 就无形中进行了类型转换。

  1. 创建变色龙类型

与上面代码类似，只是我们手动指定一个 type 值，比如：

struct Valriant {
    char type;
    union {
        int iVal;
        long lVal;
        float fVal;
    }
}

这个就是指定了要使用的类型，有什么用途呢？比如 A 和 B 通信，使用了这样一个 struct，那么 A 在给 B 传递数据的时候，可以指定当前存储的值的类型。当 B 收到结构体的时候，访问 Valriant.type，如果结果为 int，那么 B 就访问 Valriant.ival，如果值为 float，那么 B 就访问 Valriant.fVal……

位域操作，就是指定使用精度到 bit，而不是字节。比如：

struct A {
    unsigned char a:1;
    char b:3;
    unsigned char c:1;
    unsigned char d:1;
    unsigned char e:2;
};

这里整个 sizeof(A) 的值是多少呢，为 1，因为总共占用 1+3+1+2+1 个比特。

一般不太用得着位域，但是在位操作的时候，比较方便，尤其在嵌入式系统上，需要操作按 bit 位来操作的话，有时候还是挺好使的。

内存布局初探

首先，看一下一段 c++ 中的代码：

class A {};
class B {
    int a;
    fun();
}
class C (
    int a;
    virtual fun();
)

sizeof(A)、sizeof(B)、sizeof(C) 分别是多少呢？

分别为 1、4、不确定但起码大于 8，第二个为 4 是因为代码段不占用空间，第三个和虚函数的概念有关。看不明白没关系，有一本书《深度探索 C++ 对象模型》。

下面再来说说Linux的虚拟内存管理：

Linux 的虚拟内存管理机制为应用程序和驱动程序提供了两种服务：

  • 使每个进程拥有自己独立的内存地址空间;对于 32 位 Linux 来说，每个每个任务可寻址的内存地址空间都为 0x00000000 ~ 0xFFFFFFFF(4GB）
  • 当物理内存不够 4GB 时，虚拟内存管理模块会用外存空间模拟内存空间，并且该模拟过程对应用程序是透明的。

什么是用户地址空间，什么是内核地址空间呢？

  • Linux 将每个进程的 4GB 的独立地址空间又划分为用户地址空间（0x00000000 ~ 0xBFFFFFFF）和内核地址空间（0xC0000000 ~ 0xFFFFFFFF）两部分。
  • 操作系统内核代码和数据存放在内核地址空间；每个进程自己私有的代码和数据存放在用户地址空间。
  • 虽然 Linux 的内核代码和数据被映射到了每个进程的地址空间中（所有进程看到的内容是相同的），但在实际的物理内存中，只有内核代码和数据的一份拷贝。
  • 随着进程的切换，当前有效的用户地址空间也在切换。

随着进程上下文的切换，内核进程地址空间保持不变，变的是用户地址空间。

C 程序的运行时布局，其中存在着代码区、常量数据区、静态数据区、堆区以及栈区等，具体的内容后面再说，这里先看一个有趣的例子：

int add(int x, int y){
    return x+y;
}
typedef int (*FP)(int,int);
int main(){
    unsigned char buff[1024];
    unsigned char *ps = (unsigned char*)add, *pd=buff;
    void *pp = buff;
    while(1){
        *pd = *ps;
        if(*ps == 0xc3) break;
        ps++;pd++;
    }
    FP fp = (FP)pp;
    printf("3+5=%d\n", fp(3,5));
    return 0;
}

代码的第四行定义了一个函数指针 FP，指向的是一个参数为 (int, int) 的，返回值为 int 型的函数。比如 add。

这段代码的作用就是将代码区的内容拷贝到 buff 中，然后用函数指针指向 buff，把 buff 的数据再当作函数代码来执行。在高手的眼里，代码和数据，都是存储。

至于第 11 行的 0xc3 的由来，则是因为，在汇编代码中，每一个函数的最后一条机器指令都是不带参数的 ret，即为 0xc3。

更多的关于 C 程序的代码布局问题将在后面的课程学习。

main 函数为空输出 hello, world

在 c++ 中，这个很容易实现，直接使用类的构造函数即可。

在 c 中呢，也可以的，使用的还是神奇的 __attribute__ 关键字，声明 construct 以及 destructor，在 gcc 中，如下代码：

__attribute((constructor)) void before_main(){
    ......
}
__attribute((destructor)) coid after_main()
{
   ......     
}
int main(){
    ......
}

这里就声明了分别用于 main 开始前和 main 结束后运行的函数。

另外也可以使用 atexit 函数来注册结束函数。并且 atexit 是标准的 C 语言函数。atexit 通常用于 main 函数结束后进行一些清理工作，比如资源回收。为什么不在 main 函数 return 前进行这些才做呢？考虑到 main 函数是有可能异常退出的，这个时候还没等运行到 return 进程就结束了，也就无法进行资源回收等操作，但是如果是用 atexit 注册的函数，在这种情况下，也能运行进行操作。

《程序员的自我修养》这本书中有讲装载器，装载器在装载的时候，有一个表格，其中就记录了哪些函数在 main 前运行，哪些函数在 main 后运行。在程序的静态内存布局 ELF 中，.init 段记录了在 main 函数前运行的代码，.finit 记录了在 main 函数之后运行的代码。
